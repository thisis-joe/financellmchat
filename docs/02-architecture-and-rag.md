# 02. 구조와 RAG 원리

이 문서는 코드가 왜 Spring, Python, MySQL, 프론트로 나뉘었는지 설명한다.  
핵심은 `웹 백엔드로서의 Spring`과 `검색/생성 파이프라인으로서의 Python RAG`를 분리해서 이해하는 것이다.

## 주제와 범위

프로젝트 주제는 `BNK부산은행 상품공시 > 예금상품 > 적립식예금` PDF 기반 RAG다. 부산은행 상품공시는 예금, 대출, 기타상품, 서비스수수료까지 넓지만, 현재는 사용자가 제공한 적립식예금 PDF만 실제 근거로 사용한다.

범위를 좁힌 이유는 단순하다. 처음부터 모든 상품군을 넣으면 PDF 분류, 표 추출, 검색 품질, 답변 검증 문제가 한꺼번에 터진다. 지금은 작은 범위에서 끝까지 동작하고 설명 가능한 구조를 만드는 것이 우선이다.

## 전체 요청 흐름

```text
브라우저 챗봇
  -> Spring Boot /api/ask
  -> RagService
  -> RagClient
  -> Python FastAPI /ask
  -> 질문 유형 분류
  -> 직접 응답 또는 RAG 검색
  -> LLM 생성 또는 문서기반 fallback
  -> Spring이 query_histories 저장
  -> 사용자가 답변근거/피드백 확인
```

Spring은 단순 프록시가 아니다. HTTP 요청, DTO 검증, Python 호출, 질문 이력 저장, 피드백 저장, PDF 다운로드, 예외 응답, 정적 프론트 제공을 담당한다. Python은 PDF chunk 검색, 질문 유형 분류, 검색 근거 기반 답변 생성을 담당한다.

이 분리는 Spring MVC 역량을 보여주기에도 중요하다. AI 기능이 있더라도 서비스는 결국 안정적인 백엔드 위에서 돌아간다. 질문 이력과 피드백을 DB에 남겨야 검색 품질 개선과 LoRA 데이터셋 생성까지 이어질 수 있다.

## 데이터 흐름

```text
원본 PDF
  -> data/raw
  -> PDF importer
  -> PyMuPDF 텍스트 추출
  -> chunk 분리
  -> MySQL financial_documents 저장
  -> Python RAG 검색
  -> Spring API 응답
```

핵심 테이블은 세 개다.

```text
financial_documents
  PDF chunk 지식 베이스

query_histories
  질문, 답변, 근거 JSON, 상태 저장

answer_feedbacks
  특정 질문 이력에 대한 사용자 평가 저장
```

`financial_documents`에는 제목, 카테고리, 금융기관, 상품명, 상품유형, 출처, 본문을 저장한다. 같은 상품에도 상품설명서, 약관, 특약, 변경 공시가 따로 존재할 수 있기 때문에 상품명과 제목을 분리한다.

## 질문 유형 분류

모든 입력을 RAG 검색으로 보내면 챗봇이 어색해진다. `ㅎㅇ` 같은 인사말이 청년상품 문서와 매칭되고, `예금 종류` 같은 분류 질문이 약관 문장으로 흘러갈 수 있다.

그래서 검색 전에 입력 성격을 나눈다.

```text
인사, 감사, 종료, 무의미 입력
  -> 자연스러운 안내 답변

예금 종류, 적립식예금 목록
  -> 직접 응답

나이, 군인, 고령층 같은 조건
  -> 자연스러운 상황 응답 또는 추천 흐름

상품명 + 설명/가입대상/금리/서류/해지 질문
  -> RAG 검색
```

현재 원칙은 이 분류를 자연스럽게 유지하는 것이다. 특정 케이스만 맞추는 것이 아니라, 기본적으로 GPT처럼 대화하고 필요할 때 RAG로 유도한다.

## 검색 방식

현재 검색은 세 가지를 섞는다.

```text
TF-IDF 점수
+ embedding 의미 유사도
+ 상품명/질문 의도 규칙 보정
= 최종 검색 점수
```

TF-IDF는 정확한 단어와 상품명에 강하다. embedding은 `혜택 받으려면`, `누가 만들 수 있어`, `만기 때 필요한 것`처럼 문서 표현과 다른 자연어 질문에 강하다. 금융 상품 검색에서는 둘 중 하나만 쓰기보다 함께 쓰는 편이 안전하다.

상품명이 질문에 들어오면 다른 상품 문서에는 패널티를 준다. 예를 들어 `펫 적금 혜택`이라고 물었는데 청년상품이 섞이면 답변 신뢰도가 떨어지기 때문이다.

## 추천 흐름

추천은 일반 검색과 다르다. 질문과 가장 비슷한 chunk 하나를 찾는 문제가 아니라, 사용자 조건과 맞지 않는 상품을 제외하고 후보를 설명하는 문제다.

현재 추천 흐름은 다음처럼 작게 유지한다.

```text
질문에서 나이와 목적 감지
  -> 상품 PDF의 가입대상/가입자격/가입제한 구간 확인
  -> 사용자 조건과 맞지 않는 상품 제외
  -> 목적 힌트가 있으면 관련 상품 우선
  -> 후보와 이유 설명
  -> 근거와 PDF 다운로드는 답변근거로 제공
```

예를 들어 10대나 만 18세 사용자는 PDF상 `만 19세 이상` 조건이 있는 청년도약계좌를 추천 후보에서 제외해야 한다. 50대 사용자는 청년 전용 상품보다 연령 제한이 없거나 중장년층 조건에 맞는 상품을 먼저 봐야 한다.

다만 현재 검증은 나이 조건처럼 문장 패턴이 비교적 명확한 항목을 우선 처리한다. 소득, 거주지, 무주택, 군복무, 기관 선정 여부처럼 여러 조건이 결합된 자격은 아직 완전 자동 판정하지 않는다.

## 답변 생성

답변 생성에는 세 가지 경로가 있다.

```text
1. 로컬 Gemma MLX
   -> 무료, 구조 학습에 좋음
   -> 한국어 대화 유연성은 한계가 있음

2. 문서기반 fallback
   -> 모델 실패 시에도 안전하게 답변
   -> 자연스러운 말투는 부족할 수 있음

3. GPT API
   -> 가장 자연스러운 대화와 상담 말투 기대
   -> API 키와 비용 관리 필요
```

중요한 원칙은 동일하다. 모델은 상품 조건을 외워서 답하면 안 된다. RAG가 찾은 문서 근거를 읽고, 그 근거 안에서만 답변해야 한다.

현재 답변 원칙은 본문을 양식처럼 보이지 않게 만드는 것이다. `핵심 답변`, `확인 내용`, `출처` 같은 제목은 답변 본문에서 줄이고, 사용자는 자연스러운 문장으로 먼저 답을 본다. 검색 score, snippet, PDF 다운로드는 작은 `답변근거` 버튼으로 확인한다.

코드에서는 생성 모드를 환경변수로 바꿀 수 있다. `RAG_GENERATION_MODE=mlx`는 로컬 Gemma MLX 경로이고, `RAG_GENERATION_MODE=openai`는 OpenAI Responses API를 호출하는 경로다. OpenAI 경로는 `OPENAI_API_KEY`가 있을 때만 동작하며, 키가 없거나 호출이 실패하면 문서기반 fallback으로 돌아간다.

## GPT API 선택 모드

GPT API는 RAG 검색을 대체하지 않는다. 역할은 검색된 근거를 더 자연스러운 상담 답변으로 바꾸는 것이다.

흐름:

```text
사용자 질문
  -> Python RAG 검색
  -> 검색된 chunk와 사용자 조건 정리
  -> GPT API에 질문 + 근거 전달
  -> 자연스러운 답변 생성
  -> Spring이 이력과 근거 저장
```

이 방식은 모바일 시연에서 특히 유리하다. 사용자는 내부 검색 구조보다 “말이 자연스럽고 내 상황을 이해하는지”를 먼저 느끼기 때문이다.

## LangGraph의 위치

LangGraph는 현재 자율 에이전트라기보다 RAG 단계를 명확히 연결하는 워크플로 도구다.

```text
load_documents
  -> retrieve_documents
  -> generate_answer
```

초급 프로젝트에서는 LangGraph를 복잡한 에이전트로 쓰기보다, 상태가 단계별로 어떻게 바뀌는지 보는 편이 더 좋다. 나중에 질문 재작성, 검색 재시도, 답변 검증 노드를 추가하면 에이전트 워크플로로 확장할 수 있다.

## Spring MVC 책임

Spring에서 강조할 수 있는 역량은 다음이다.

- Controller는 HTTP 입출력만 맡긴다.
- Service는 Python 호출, 이력 저장, 트랜잭션 경계를 관리한다.
- Repository는 JPA로 DB 접근을 담당한다.
- Python RAG 실패 시 fallback 경로를 둔다.
- 질문 이력과 피드백을 연결해 AI 품질 개선 데이터로 남긴다.
- PDF 다운로드는 외부 링크가 아니라 저장된 원본 파일을 attachment로 내려준다.

이 프로젝트는 MSA처럼 보이기 위해 억지로 서비스를 나누지 않는다. 먼저 Spring API와 Python RAG API의 계약을 명확히 하고, 나중에 PDF importer, RAG API, 피드백 도메인을 분리할 수 있는 구조로 둔다.

## 현재 한계

- 표가 이미지로 들어간 PDF는 PyMuPDF만으로 읽지 못할 수 있다.
- 추천 자격 판정은 나이 조건 중심이고, 소득/거주지/무주택 같은 복합 조건은 아직 제한적이다.
- 로컬 Gemma는 무료지만 자연스러운 한국어 상담 품질에는 한계가 있다.
- 개인 Mac이 꺼지면 Cloudflare Tunnel API도 멈춘다.
- 현재 문서 수에서는 메모리 embedding 캐시가 충분하지만, 문서가 많아지면 vector index가 필요할 수 있다.

## 용어 메모

- TF-IDF: 특정 단어가 문서에서 얼마나 중요한지 계산하는 전통적 검색 방식이다.
- embedding: 문장을 의미를 담은 숫자 벡터로 바꾸는 방식이다.
- hybrid search: 키워드 검색과 의미 검색을 섞는 방식이다.
- PyMuPDF: PDF에서 텍스트를 추출하는 Python 라이브러리다.
- LangGraph: 작업 단계를 상태 그래프로 연결하는 도구다.
- MLX: Apple Silicon에서 모델 실행을 최적화하는 런타임이다.
- GPT API: OpenAI 모델을 서버에서 호출해 답변을 생성하는 방식이다.
- fallback: 주 경로가 실패할 때 대체 경로로 응답하는 방식이다.
- 트랜잭션: 여러 DB 작업을 하나의 성공/실패 단위로 묶는 방식이다.
- attachment: 브라우저가 파일을 열지 않고 다운로드하도록 하는 HTTP 응답 방식이다.
- vector index: 많은 embedding 중 가까운 벡터를 빠르게 찾기 위한 검색 구조다.
