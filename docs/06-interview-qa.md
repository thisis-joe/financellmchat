# 06. 면접 예상 질문과 답변

이 문서는 면접에서 프로젝트를 설명할 때 사용할 질문과 답변을 정리한다.  
단순히 기술 이름을 아는지보다, 문제를 어떻게 발견했고 어떤 기준으로 해결했는지를 말하는 데 초점을 둔다.

질문은 프로젝트에서 실제로 나올 수 있는 범위를 최대한 넓게 잡았다. Spring MVC, DB CRUD, RAG, AI, LoRA/QLoRA뿐 아니라 예외 처리, 배포, 모바일 시연, 금융 답변 안전성, 운영 확장까지 포함한다. 면접에서는 모든 답변을 외우기보다, 각 질문의 문제 발견 → 판단 기준 → 해결 흐름을 자기 말로 설명하는 것이 중요하다.

## 1. 이 프로젝트를 한 문장으로 설명해 보세요.

이 프로젝트는 BNK부산은행의 `예금상품 > 적립식예금` 상품공시 PDF를 MySQL에 저장하고, 사용자의 질문에 맞는 문서를 검색한 뒤 자연스러운 답변을 생성하는 Spring Boot 기반 RAG 챗봇입니다.

처음에는 단순 문서 검색 서비스처럼 시작했지만, 실제 질문을 넣어 보니 `ㅎㅇ`, `예금 종류`, `50대 추천`처럼 검색으로만 처리하면 안 되는 입력이 많았습니다. 그래서 Spring은 API, 이력 저장, 피드백, 다운로드를 담당하고, Python RAG 서비스는 문서 검색과 답변 생성을 담당하도록 나누었습니다. 현재는 로컬 Gemma, 문서기반 fallback, 선택형 OpenAI API 생성 모드를 모두 고려할 수 있는 구조입니다.

## 2. 왜 Spring Boot와 Python RAG 서비스를 분리했나요?

Spring Boot는 웹 백엔드의 안정적인 요청 처리에 강하고, Python은 RAG, embedding, LLM 라이브러리 사용에 강합니다. 그래서 역할을 나누었습니다.

처음에는 모든 것을 Spring 안에서 처리할 수도 있었지만, PDF 검색, embedding 모델, LangGraph, 로컬 LLM 실행은 Python 생태계가 훨씬 자연스러웠습니다. 반대로 사용자 요청, DB 트랜잭션, 질문 이력 저장, 피드백 저장, PDF 다운로드는 Spring MVC 구조로 다루는 것이 좋았습니다.

문제는 서버 간 통신이었습니다. Spring에서 FastAPI를 호출할 때 JSON body와 `Content-Type`이 맞지 않아 `422`가 발생했습니다. 이 문제는 `RagClient`에서 DTO를 명확히 JSON으로 직렬화하고 `application/json`을 지정해 해결했습니다. 이 경험을 통해 외부 AI 서비스를 붙일 때도 HTTP 계약을 명확히 잡아야 한다는 점을 배웠습니다.

## 3. Spring MVC 구조에서 Controller, Service, Repository를 어떻게 나눴나요?

Controller는 HTTP 요청과 응답만 담당하게 했고, Service는 비즈니스 흐름과 트랜잭션 경계를 담당하게 했습니다. Repository는 JPA를 통해 DB 접근만 맡습니다.

예를 들어 `/api/ask` 요청이 들어오면 Controller는 요청 DTO를 받고 Service로 넘깁니다. Service는 Python RAG 호출, fallback 처리, 질문 이력 저장을 담당합니다. 이력 저장 뒤에는 `historyId`를 반환해서 사용자가 나중에 `답변근거`나 피드백을 남길 수 있게 했습니다.

이렇게 나눈 이유는 유지보수 때문입니다. 만약 Controller 안에서 Python 호출, DB 저장, 예외 처리를 모두 하면 기능이 늘어날수록 테스트와 수정이 어려워집니다. 실제로 피드백 기능과 답변근거 API를 추가할 때 Service 중심 구조 덕분에 기존 API 흐름을 크게 흔들지 않고 확장할 수 있었습니다.

## 4. DB CRUD는 이 프로젝트에서 어떻게 쓰였나요?

이 프로젝트에서 CRUD는 단순 게시판 CRUD가 아니라 RAG 서비스 운영 데이터 관리에 사용됩니다.

- Create: PDF chunk를 `financial_documents`에 저장하고, 질문 이력을 `query_histories`에 저장합니다.
- Read: 질문 시 문서 chunk를 조회하고, 답변근거 API에서 저장된 citation을 다시 읽습니다.
- Update: 현재 핵심 흐름에는 많지 않지만, 피드백이나 향후 문서 재처리 상태를 갱신할 수 있는 구조입니다.
- Delete: PDF를 다시 적재할 때 기존 chunk를 삭제하고 새로 넣는 `--replace` 흐름이 있습니다.

특히 `query_histories`는 중요합니다. 단순히 답변을 사용자에게 보여주고 끝내는 것이 아니라, 질문, 답변, 검색 근거, 상태를 저장합니다. 이 데이터가 있어야 나중에 피드백 분석, 검색 품질 개선, LoRA 학습 데이터 후보 선별까지 이어질 수 있습니다.

## 5. Python RAG가 꺼져 있으면 서비스는 어떻게 되나요?

Python RAG가 꺼져도 Spring 전체가 바로 죽지 않도록 fallback 구조를 두었습니다.

처음에는 Python 호출 실패가 곧 전체 질문 실패로 이어질 수 있었습니다. 하지만 실제 서비스에서는 외부 AI 런타임이나 모델 서버가 불안정할 수 있습니다. 그래서 `RagClient` 호출 실패 시 Spring Repository 기반 검색이나 문서기반 fallback 답변으로 돌아가게 했습니다.

이 판단은 품질을 완벽하게 보장한다는 뜻은 아닙니다. 대신 사용자에게 완전한 장애 화면을 보여주는 것보다, 제한된 답변이라도 제공하고 상태를 남기는 쪽이 운영에 더 안전하다고 봤습니다.

## 6. RAG 파이프라인은 어떤 단계로 동작하나요?

현재 RAG 파이프라인은 크게 세 단계입니다.

```text
load_documents
  -> retrieve_documents
  -> generate_answer
```

`load_documents`는 MySQL에서 부산은행 적립식예금 PDF chunk를 읽습니다. `retrieve_documents`는 질문과 관련 있는 chunk를 찾습니다. `generate_answer`는 검색된 근거를 기반으로 로컬 Gemma, OpenAI API 선택 모드, 또는 문서기반 fallback으로 답변을 만듭니다.

LangGraph는 이 흐름을 상태 그래프로 연결하기 위해 사용했습니다. 여기서 LangGraph를 복잡한 자율 에이전트로 쓰기보다, RAG 단계가 어떤 순서로 진행되고 상태가 어떻게 바뀌는지 명확히 보여주는 워크플로 도구로 사용했습니다.

## 7. 검색 품질 문제는 어떻게 발견하고 개선했나요?

처음에는 TF-IDF 검색만 사용했습니다. 구현은 단순했지만 `펫 적금 혜택 받으려면?` 같은 자연어 질문에서 원하는 우대이율 페이지가 아니라 가입대상 페이지가 먼저 나오는 문제가 있었습니다.

문제는 두 가지였습니다. 첫째, 상품명은 맞아도 문서 안의 항목이 맞지 않았습니다. 둘째, 사용자는 `혜택`이라고 묻지만 문서에는 `우대이율`이라고 적혀 있는 표현 차이가 있었습니다.

그래서 검색을 세 단계로 개선했습니다.

1. 상품명 감지: `펫 적금`처럼 명시된 상품이 있으면 다른 상품 문서에 패널티를 줍니다.
2. 질문 의도 감지: 가입대상, 금리, 서류, 해지 같은 의도를 구분하고 관련 항목명에 점수를 더합니다.
3. embedding 검색: 자연어 표현과 문서 표현이 달라도 의미가 가까운 chunk를 찾도록 `multilingual-e5-small` embedding을 추가했습니다.

결과적으로 TF-IDF, embedding, 규칙 기반 재점수화를 섞은 hybrid 검색이 되었습니다.

## 8. 왜 모든 질문을 RAG 검색으로 보내지 않았나요?

모든 질문을 RAG로 보내면 챗봇이 어색해집니다. 실제로 `ㅎㅇ` 같은 인사말이 청년상품 문서와 매칭되거나, `예금 종류` 질문이 약관 문서로 흘러가는 문제가 있었습니다.

그래서 검색 전에 질문 유형을 나눴습니다.

- 인사, 감사, 종료, 무의미 입력은 직접 응답
- 예금 종류, 상품 목록은 직접 안내
- 추천 질문은 추천 전용 흐름
- 상품명, 가입대상, 금리, 서류, 설명 질문은 RAG 검색

이 구조를 통해 사용자는 GPT처럼 자연스러운 응답을 먼저 받고, 금융상품 정보가 필요한 경우에만 RAG 근거 기반 답변을 받게 됩니다.

## 9. 추천 질문에서 10대에게 청년도약계좌를 추천하지 않도록 어떻게 처리했나요?

처음에는 상품명이나 키워드만 보고 추천했기 때문에 10대에게도 청년도약계좌가 추천될 수 있었습니다. 그런데 청년도약계좌는 PDF상 만 19세 이상 조건이 있습니다.

그래서 추천 전에 각 상품 PDF의 `가입대상`, `가입자격`, `가입제한` 구간을 읽고 나이 조건을 추출하도록 했습니다. `만 19세 이상`, `만 34세 이하`, `연령에 관계없이`, `제한없음` 같은 표현을 정규식으로 읽어 사용자 나이와 비교합니다.

이 방식은 완전한 금융 자격 판정은 아닙니다. 소득, 거주지, 무주택, 기관 선정 여부 같은 복합 조건은 아직 제한적입니다. 하지만 적어도 명확한 나이 조건을 어기는 추천은 막을 수 있게 되었습니다.

## 10. 답변근거 API는 왜 만들었나요?

RAG에서 중요한 것은 답변뿐 아니라 왜 그 답변이 나왔는지 확인하는 것입니다. 특히 금융 상품공시는 근거가 틀리면 답변이 그럴듯해도 위험합니다.

그래서 답변 시점의 citation을 `query_histories`에 JSON으로 저장하고, `GET /api/histories/{historyId}/evidence`로 다시 확인할 수 있게 했습니다. 이 API는 새 검색을 다시 돌리지 않고, 당시 답변에 사용된 근거를 그대로 보여줍니다.

프론트에서는 이 기능을 크게 노출하지 않고 작은 `답변근거` 버튼으로 만들었습니다. 일반 사용자는 자연스러운 답변을 먼저 보고, 필요할 때만 score, snippet, PDF 다운로드를 확인할 수 있습니다.

## 11. 왜 PDF 원문 링크 대신 다운로드로 바꿨나요?

처음에는 출처 카드에 `원문` 링크를 두었지만, 실제 링크가 개별 PDF가 아니라 유효하지 않은 공시 목록 주소로 이어지는 문제가 있었습니다. 사용자가 클릭했는데 원하는 문서가 열리지 않으면 신뢰도가 떨어집니다.

그래서 citation의 `documentId`를 기준으로 DB chunk를 찾고, 원본 PDF 파일을 서버에서 attachment로 내려주는 다운로드 API를 만들었습니다. 화면에서도 `원문` 대신 `다운로드`라고 표시했습니다.

이 방식은 사용자가 실제 근거 PDF를 받을 수 있고, 외부 사이트 링크 구조가 바뀌어도 시연 흐름이 깨지지 않는 장점이 있습니다.

## 12. 로컬 Gemma와 OpenAI API를 왜 둘 다 고려했나요?

로컬 Gemma는 무료로 실행할 수 있고, 모델 실행 구조를 직접 이해하기 좋습니다. 하지만 한국어 상담 말투, 짧은 입력의 의도 해석, 자연스러운 문장 생성에는 한계가 있습니다.

OpenAI API는 비용과 API 키 관리가 필요하지만, 자연스러운 답변 품질이 좋습니다. 그래서 기본 시연은 로컬 Gemma와 fallback으로 가능하게 두고, 필요할 때 `RAG_GENERATION_MODE=openai`로 전환할 수 있는 선택 모드를 추가했습니다.

중요한 점은 OpenAI API도 RAG를 대체하지 않는다는 것입니다. 상품 조건은 PDF 검색으로 가져오고, OpenAI API는 검색된 근거를 읽기 좋은 상담 답변으로 조립하는 역할입니다.

## 13. LoRA와 QLoRA는 이 프로젝트에서 어디에 쓰이나요?

LoRA/QLoRA는 상품 지식을 외우게 하는 용도가 아닙니다. 금융 상품 조건은 바뀔 수 있기 때문에 모델 파라미터에 외우게 하면 위험합니다.

이 프로젝트에서 LoRA/QLoRA의 역할은 RAG 근거를 읽고 자연스럽고 안전하게 답변하는 습관을 학습시키는 것입니다. 예를 들어 `출처를 본문에 길게 늘어놓지 않기`, `조건이 맞지 않는 상품을 조심스럽게 제외하기`, `근거에 없는 내용은 단정하지 않기` 같은 응답 정책을 학습시킬 수 있습니다.

학습 데이터는 `query_histories`, `citationsJson`, `answer_feedbacks`에서 출발합니다. 좋은 답변은 후보로 삼고, 나쁜 답변은 사람이 고쳐 JSONL messages 포맷으로 만듭니다.

## 14. LoRA와 QLoRA의 차이를 설명해 보세요.

LoRA는 원본 모델 전체를 다시 학습하지 않고 작은 adapter만 학습하는 방식입니다. 비용이 낮고 원본 모델을 보존할 수 있습니다.

QLoRA는 양자화된 모델 위에서 LoRA adapter를 학습합니다. 더 큰 모델을 제한된 GPU 메모리에서 실험할 때 유리합니다. 데이터 포맷은 거의 같고, 차이는 학습 런타임과 메모리 전략입니다.

이 프로젝트에서는 먼저 LoRA로 작은 실험을 하고, 모델 크기나 메모리 한계가 생기면 QLoRA로 확장하는 순서가 맞다고 봅니다.

## 15. Spring 역량을 보여줄 수 있는 부분은 무엇인가요?

이 프로젝트에서 Spring은 단순히 Python API를 호출하는 껍데기가 아닙니다.

- Controller, Service, Repository 계층 분리
- Python RAG 호출과 fallback 처리
- 질문 이력 저장
- 피드백 저장
- 답변근거 조회
- PDF 다운로드 API
- JPA Entity와 Repository 설계
- 예외 상황에서 사용자 흐름이 멈추지 않도록 처리

특히 질문 이력과 피드백을 저장하는 구조는 AI 서비스에서 중요합니다. 답변 품질을 시스템으로 회수하고, 나중에 검색 개선이나 LoRA 학습 데이터로 이어질 수 있기 때문입니다.

## 16. 이 프로젝트의 한계는 무엇이고, 어떻게 개선할 건가요?

현재 한계는 명확합니다.

첫째, PDF 표가 이미지로 들어간 경우 PyMuPDF만으로 읽지 못할 수 있습니다. 이 경우 OCR 도입이 필요합니다. 다만 처음부터 OCR을 붙이면 복잡도가 커지므로, 실제 누락 사례를 확인한 뒤 도입하는 것이 맞다고 판단했습니다.

둘째, 추천 조건 검증은 이제 `product_knowledge.json` 비교 지식표를 사용하지만, 소득, 무주택, 거주지, 군복무 상태 같은 복합 조건은 아직 완전 자동 판정하지 않습니다. 다음 단계에서는 이 복합 조건의 추출 정확도를 높여야 합니다.

셋째, 로컬 모델의 자연어 품질은 한계가 있습니다. 그래서 OpenAI API 선택 모드를 추가했고, 장기적으로는 좋은 답변 로그를 LoRA/QLoRA 학습 데이터로 만들어 로컬 모델의 응답 품질을 개선할 계획입니다.

넷째, 현재 배포는 개인 Mac과 Cloudflare Tunnel에 의존합니다. 실제 운영이라면 Spring API, Python RAG, MySQL, PDF 저장소를 클라우드 환경으로 옮기고 모니터링과 장애 대응을 강화해야 합니다.

## 17. 실무에 투입된다면 이 경험을 어떻게 활용할 수 있나요?

이 프로젝트를 통해 단순히 AI 모델을 호출하는 것보다, AI 기능을 실제 백엔드 서비스 안에 안전하게 넣는 흐름을 경험했습니다.

Spring MVC에서는 요청 흐름, 계층 분리, DB 저장, 예외 처리, 다운로드 API, 피드백 API를 구현했습니다. RAG에서는 PDF 적재, chunking, 검색 품질 개선, embedding, 질문 라우팅, 답변 생성, 근거 확인을 다뤘습니다. AI 쪽에서는 로컬 LLM, OpenAI API 선택 모드, LoRA/QLoRA 학습 데이터 설계까지 연결했습니다.

실무에서는 새로운 기술을 붙이는 것보다 도입 경로와 실패 대응이 중요하다고 봅니다. 이 프로젝트에서도 한 번에 거대한 AI 시스템을 만든 것이 아니라, 작은 Spring MVC 서비스에서 시작해 검색, 생성, 피드백, 배포, 학습 데이터까지 단계적으로 확장했습니다.

## 18. 왜 처음부터 MSA로 만들지 않았나요?

처음부터 MSA로 나누면 기술적으로는 그럴듯해 보일 수 있지만, 현재 단계에서는 도메인 경계와 트랜잭션 경계가 충분히 검증되지 않았습니다.

이 프로젝트에서는 먼저 Spring Boot 단일 백엔드와 Python RAG 서비스의 경계만 명확히 했습니다. 질문 API, 피드백, 문서 다운로드, RAG 호출이 어떤 식으로 연결되는지 안정화한 뒤에야 어떤 부분을 분리할지 판단할 수 있습니다.

MSA로 확장한다면 PDF importer, RAG 검색/생성 API, 질문 이력/피드백 도메인을 후보로 봅니다. 하지만 분리 전에 API 계약, 장애 대응, 데이터 정합성 책임, 회귀 테스트 시점을 먼저 정해야 합니다.

## 19. 이 프로젝트에서 트랜잭션 경계는 어디에 두었나요?

Spring Service 계층이 트랜잭션 경계를 갖는 것이 맞다고 판단했습니다. 질문을 받고, Python RAG를 호출하고, 응답과 citation을 `query_histories`에 저장하는 흐름이 하나의 서비스 작업이기 때문입니다.

다만 외부 Python 호출 자체는 DB 트랜잭션 안에 길게 묶는 것이 항상 좋은 것은 아닙니다. 외부 호출이 오래 걸리면 DB 커넥션을 오래 점유할 수 있기 때문입니다. 현재 프로젝트는 학습용이라 구조를 단순하게 유지했지만, 실무에서는 외부 호출과 DB 저장의 경계를 더 세밀하게 나누거나 비동기 처리도 검토할 수 있습니다.

## 20. API 응답 DTO를 왜 따로 만들었나요?

Entity를 그대로 API 응답으로 내보내면 DB 구조가 외부 계약이 되어 버립니다. 그래서 요청/응답 DTO를 분리하는 것이 안전합니다.

이 프로젝트에서는 RAG 응답, citation, evidence, feedback 요청이 각각 목적이 다릅니다. 예를 들어 `RagEvidenceResponse`는 사용자가 본 답변의 근거를 보여주기 위한 응답이고, `AskResponse`는 실제 질문 답변을 위한 응답입니다. DTO를 분리하면 화면 요구사항이 바뀌어도 Entity를 직접 흔들지 않을 수 있습니다.

## 21. JPA Entity 설계에서 중요하게 본 점은 무엇인가요?

`FinancialDocument`는 PDF chunk를 저장하는 지식 베이스이고, `QueryHistory`는 사용자 질문과 답변을 저장하는 이력입니다. `AnswerFeedback`은 특정 질문 이력에 연결됩니다.

중요하게 본 점은 “문서 데이터”와 “사용자 상호작용 데이터”를 분리하는 것이었습니다. 문서 chunk는 검색 대상이고, 질문 이력과 피드백은 품질 개선 대상입니다. 이 둘을 분리해야 나중에 검색 평가, 피드백 분석, LoRA 데이터셋 생성으로 이어질 수 있습니다.

## 22. N+1 문제나 성능 문제는 어떻게 볼 수 있나요?

현재 프로젝트는 대량 연관 조회가 많은 구조는 아니어서 N+1 문제가 크게 드러나지는 않습니다. 하지만 피드백 목록에서 `QueryHistory`를 함께 많이 조회하거나, 문서와 관련된 메타데이터를 복잡하게 연결하면 N+1이 발생할 수 있습니다.

실무에서는 SQL 로그를 확인하고, 필요하면 fetch join, EntityGraph, DTO projection을 사용합니다. 이 프로젝트에서는 성능 최적화보다 흐름 이해가 우선이었지만, 피드백 조회나 관리자 화면이 생기면 N+1을 점검해야 합니다.

## 23. 테스트는 어떤 기준으로 만들었나요?

테스트는 두 종류로 봤습니다. 하나는 Spring MVC API가 정상 동작하는지 보는 테스트이고, 다른 하나는 대표 질문이 계속 통과하는지 보는 회귀 테스트입니다.

AI 답변은 매번 문장이 조금 달라질 수 있으므로 완전히 같은 문장을 기대하면 테스트가 깨지기 쉽습니다. 그래서 대표 질문 스크립트는 핵심 단어, status, citation product가 맞는지를 확인합니다. 예를 들어 `10대 추천`에서는 청년도약계좌가 citation에 포함되지 않아야 합니다.

이 방식은 완벽한 평가셋은 아니지만, 프로젝트가 작을 때는 가장 중요한 흐름이 깨졌는지 빠르게 확인할 수 있습니다.

## 24. 대표 질문 46개는 왜 필요한가요?

RAG 서비스는 코드가 조금만 바뀌어도 검색 순위나 답변 형식이 흔들릴 수 있습니다. 그래서 시연에서 반드시 맞아야 하는 질문을 대표 질문으로 고정했습니다.

대표 질문에는 인사, 무의미 입력, 종료 인사, 예금 종류, 상품 목록, 10대/50대/60대 추천, 30대 직장인 추천, 펫 적금 설명, 장병내일준비적금 서류, 청년도약계좌 가입대상, 우대금리 높은 순서, 6개월/2개월 기간 질문, 소액 추천, 전체 목록, 5개 추천, 후속 맥락 질문, 고액 예치, 불만 응대, 결혼/임산부 상황 질문이 포함됩니다. 이 질문들이 통과하면 최소한 챗봇의 핵심 흐름은 유지된다고 볼 수 있습니다.

## 25. PDF chunk 크기는 어떻게 정했나요?

PDF 전체를 하나의 문서로 저장하면 검색 결과가 너무 뭉뚱그려지고, 질문과 관련 있는 작은 구간이 묻힙니다. 반대로 chunk가 너무 작으면 문맥이 끊겨 답변 생성이 어려워집니다.

그래서 약 1,400자 단위로 chunk를 나누었습니다. 이 값은 정답이라기보다 현재 문서 수와 상품설명서 길이를 고려한 시작점입니다. 실무에서는 검색 품질 평가를 보며 chunk 크기, overlap, 섹션 기준 분할을 조정해야 합니다.

## 26. PDF 표 안의 이미지 텍스트는 어떻게 처리할 건가요?

현재는 PyMuPDF로 텍스트 레이어를 추출합니다. 이 방식은 빠르고 단순하지만, 표가 이미지로 들어간 PDF는 읽지 못할 수 있습니다.

처음부터 OCR을 붙이지 않은 이유는 복잡도 때문입니다. OCR은 설치, 속도, 비용, 정확도 문제가 있습니다. 현재는 먼저 텍스트 레이어 기반으로 가능한 범위를 확인하고, 실제 누락 사례가 확인되면 OCR을 도입하는 순서가 맞다고 봤습니다.

## 27. OCR을 도입한다면 어떤 점을 고민해야 하나요?

OCR을 도입하면 표 안의 글자까지 읽을 수 있지만, 오인식이 발생할 수 있습니다. 금융상품 문서에서는 숫자, 금리, 기간, 나이 조건이 중요하므로 OCR 오류가 치명적일 수 있습니다.

그래서 OCR을 넣는다면 단순히 텍스트를 추가하는 것이 아니라, OCR 결과에 confidence를 저장하고, 숫자나 금리 조건은 사람이 검수하거나 규칙 검증을 붙이는 방식이 필요합니다. 또한 원본 텍스트 추출 결과와 OCR 결과를 구분해 디버깅 가능하게 남겨야 합니다.

## 28. 왜 vector DB를 바로 쓰지 않았나요?

현재 문서 수는 많지 않아서 Python 메모리에서 embedding을 계산하고 캐시하는 방식으로 충분했습니다. 처음부터 vector DB를 붙이면 인프라와 운영 복잡도가 커집니다.

다만 문서가 수천 개 이상으로 늘어나거나, 여러 상품군과 버전별 공시를 관리해야 한다면 vector DB 또는 FAISS 같은 vector index가 필요합니다. 이 프로젝트에서는 먼저 RAG 원리와 Spring 연동을 이해하는 것이 우선이었습니다.

## 29. embedding 모델은 왜 사용했나요?

TF-IDF는 정확한 단어 매칭에는 강하지만, 표현이 달라지면 약합니다. 사용자는 `혜택 받으려면?`이라고 묻고, 문서에는 `우대이율`이라고 적혀 있을 수 있습니다.

embedding은 이런 의미적 유사성을 보완합니다. 그래서 `intfloat/multilingual-e5-small`처럼 한국어도 지원하는 다국어 검색 모델을 사용했습니다. 단, embedding만 쓰면 상품명이 비슷한 문서가 섞일 수 있어 TF-IDF와 상품명 규칙을 함께 사용했습니다.

## 30. 프롬프트 설계에서 중요하게 본 점은 무엇인가요?

가장 중요한 것은 모델이 근거 밖의 내용을 만들지 않게 하는 것입니다. 그래서 프롬프트에는 검색 근거만 사용하고, 모르는 내용은 추측하지 말라고 지시했습니다.

또 하나의 문제는 답변이 너무 템플릿처럼 보이는 것이었습니다. 처음에는 `핵심 답변`, `확인 내용`, `출처` 같은 구조를 썼지만, 사용자가 보기에는 대화가 아니라 보고서처럼 보였습니다. 그래서 현재는 본문을 자연스러운 문장으로 만들고, 출처와 다운로드는 `답변근거` 버튼에 숨기는 방향으로 바꾸었습니다.

## 31. 모델이 답변을 중간에 끊으면 어떻게 처리했나요?

로컬 Gemma가 긴 답변을 만들 때 마지막 문장이 `문의하시면`처럼 끊기는 문제가 있었습니다. 이것은 모델 크기와 출력 토큰 제한의 한계입니다.

그래서 후처리를 추가했습니다. 깨진 bullet은 `-` 형태로 정리하고, 끝나지 않은 마지막 문장은 제거합니다. 완벽한 해결은 아니지만, 모바일 챗봇 화면에서 깨진 답변이 보이는 문제를 줄일 수 있습니다.

## 32. OpenAI API 키가 없으면 어떻게 되나요?

기본 실행은 OpenAI API가 없어도 됩니다. `RAG_GENERATION_MODE=mlx`를 쓰면 로컬 Gemma로 답변을 생성하고, 모델 실패 시 문서기반 fallback으로 돌아갑니다.

`RAG_GENERATION_MODE=openai`를 선택했는데 `OPENAI_API_KEY`가 없으면 OpenAI 호출은 실패하고 fallback 답변으로 돌아갑니다. 이렇게 한 이유는 외부 API 키나 네트워크 문제 때문에 전체 서비스가 멈추면 안 되기 때문입니다.

## 33. 비용이 드는 API를 실무에서 쓸 때 어떤 점을 고려해야 하나요?

OpenAI API 같은 외부 모델은 품질이 좋지만 비용과 지연시간이 있습니다. 그래서 모든 요청을 무조건 API로 보내기보다, 인사말이나 단순 범위 안내는 직접 응답하고, 문서 근거 기반 답변이 필요한 경우에만 생성 모델을 사용하는 것이 좋습니다.

또한 max output tokens, timeout, retry, 장애 시 fallback, 사용량 로그가 필요합니다. 프로젝트에서는 선택 모드로만 붙여 비용 통제를 쉽게 했습니다.

## 34. 보안 측면에서 조심해야 할 점은 무엇인가요?

첫째, API 키는 코드에 넣으면 안 됩니다. 환경변수로 주입해야 합니다. 둘째, PDF 다운로드 API는 사용자가 요청한 documentId가 실제 허용된 문서인지 확인해야 합니다. 셋째, 외부 배포에서는 Cloudflare Tunnel과 Vercel rewrite 경로가 의도한 API로만 연결되는지 확인해야 합니다.

또한 사용자 질문과 답변 이력에는 개인정보가 들어올 수 있습니다. 실무에서는 저장 기간, 마스킹, 접근 권한, 로그 정책을 정해야 합니다.

## 35. Cloudflare Tunnel과 Vercel 구조의 장단점은 무엇인가요?

장점은 간단한 시연이 가능하다는 점입니다. 프론트는 Vercel에 무료로 올리고, API는 개인 Mac의 Spring Boot로 연결할 수 있습니다. 포트포워딩 없이 외부 접속이 가능하다는 점도 장점입니다.

단점은 개인 Mac에 의존한다는 점입니다. Mac이 꺼지거나 절전 상태가 되면 API도 멈춥니다. 실제 운영이라면 Spring API, Python RAG, MySQL, PDF 저장소를 클라우드 환경으로 옮겨야 합니다.

## 36. 프론트는 왜 단일 HTML/CSS/JS로 유지했나요?

이 프로젝트의 핵심 학습 대상은 프론트가 아니라 Spring MVC와 RAG입니다. React나 복잡한 프론트 구조를 붙이면 학습 범위가 불필요하게 커집니다.

그래서 단일 `index.html`, `style.css`, `app.js`로 챗봇 UI만 구현했습니다. 다만 모바일 시연에서 입력창이 키보드에 가리는 문제는 사용자 경험에 직접 영향을 주므로 CSS와 visual viewport 처리를 통해 개선했습니다.

## 37. 모바일 입력창 문제는 어떻게 해결했나요?

모바일에서는 키보드가 올라올 때 입력창이 브라우저 UI나 키보드에 가려질 수 있습니다. 특히 챗봇은 입력창이 항상 보여야 하기 때문에 이 문제가 큽니다.

해결 방향은 모바일에서 챗봇 패널을 전체화면에 가깝게 만들고, 헤더는 고정하고, 메시지 영역만 스크롤되게 하는 것이었습니다. 입력 영역은 visual viewport 변화에 맞춰 키보드 위에 붙도록 CSS 변수를 사용했습니다. 프론트가 핵심은 아니지만 시연 품질에는 중요한 문제였습니다.

## 38. 상품 설명 질문은 어떻게 처리하나요?

`펫 적금 설명해줘`, `청년도약계좌 알려줘` 같은 질문은 다운로드만 제공하면 안 됩니다. 사용자는 먼저 내용을 알고 싶어 합니다.

그래서 `설명`, `알려줘`, `뭐야`, `내용`, `정리` 같은 표현을 `상품설명` 의도로 감지하도록 했습니다. 이 경우 상품 개요, 특징, 가입대상, 가입기간, 가입금액 같은 문서 구간을 우선 요약합니다. PDF 다운로드는 본문이 아니라 답변근거에서 확인할 수 있습니다.

## 39. 금융 상품 추천에서 주의할 점은 무엇인가요?

금융상품 추천은 단정적으로 하면 위험합니다. 이 프로젝트는 금융 자문 서비스가 아니라 상품공시 기반 후보 안내입니다.

그래서 답변에서는 사용자의 나이와 목적에 맞는 후보를 제시하되, 최종 가입 가능 여부와 우대금리는 가입 시점, 개인 조건, 은행 확인 결과에 따라 달라질 수 있다고 안내합니다. 특히 조건이 맞지 않는 상품은 왜 제외했는지 설명하려고 했습니다.

## 40. 답변이 자연스럽다는 것과 근거 기반이라는 것은 충돌하지 않나요?

충돌할 수 있습니다. 자연스럽게 말하려다 보면 모델이 근거에 없는 내용을 덧붙일 위험이 있고, 근거만 엄격히 쓰면 답변이 딱딱해질 수 있습니다.

이 프로젝트에서는 역할을 나누어 해결했습니다. 본문은 자연스럽게 쓰되, 내용은 검색 근거 안에서만 만들도록 프롬프트와 fallback을 구성했습니다. 출처와 PDF 다운로드는 본문에 길게 넣지 않고 답변근거 버튼으로 분리했습니다.

## 41. 로그와 모니터링은 어떻게 확장할 수 있나요?

현재는 질문 이력과 citation을 DB에 저장하고, 대표 질문 스크립트로 회귀 검증을 합니다. 실무에서는 여기에 요청 시간, Python RAG 응답 시간, 검색 top score, fallback 발생 여부, OpenAI API 사용량을 로그로 남겨야 합니다.

이 데이터가 있어야 느린 요청, 검색 품질 저하, 모델 장애를 구분할 수 있습니다. 특히 RAG 서비스에서는 “답변이 이상하다”는 문제를 검색 문제, 생성 문제, 데이터 적재 문제로 나눠 볼 수 있어야 합니다.

## 42. DB에 저장된 질문 이력은 어떻게 활용할 수 있나요?

질문 이력은 단순 로그가 아닙니다. 사용자가 어떤 질문을 많이 하는지, 어떤 답변이 도움이 되었는지, 어떤 검색 근거가 자주 실패하는지 볼 수 있는 데이터입니다.

이 프로젝트에서는 `query_histories`와 `answer_feedbacks`를 연결해 LoRA/QLoRA 학습 데이터 후보를 만들 수 있게 했습니다. `HELPFUL` 답변은 좋은 후보가 되고, `NOT_HELPFUL` 답변은 사람이 수정해 정답 데이터로 만들 수 있습니다.

## 43. 실무에서 문서가 자주 바뀌면 어떻게 처리하나요?

금융 상품공시는 개정될 수 있습니다. 따라서 모델에 지식을 외우게 하면 안 되고, 원본 PDF와 적재 시점을 관리해야 합니다.

실무에서는 문서 버전, 게시일, 개정일, source 파일 해시를 저장하고, 새 문서가 들어오면 기존 문서와 비교해 재적재해야 합니다. 검색 결과에도 어떤 버전 문서에서 나온 답변인지 추적할 수 있어야 합니다.

## 44. 현재 구조에서 가장 먼저 개선하고 싶은 것은 무엇인가요?

가장 먼저 개선하고 싶은 것은 구조화된 조건 추출의 정확도입니다. 지금은 `product_knowledge.json`에 가입대상, 기간, 납입금액, 기본금리, 우대금리, 예금자보호 여부를 뽑아 추천에 사용합니다. 다음 단계에서는 소득, 거주지, 무주택, 군복무, 우대조건 비용 같은 복합 조건을 더 정교하게 뽑고 싶습니다.

그렇게 하면 추천 답변이 단순 검색 결과가 아니라 더 정확한 조건 검증을 거친 후보 안내에 가까워질 수 있습니다. 다만 이 작업은 OCR, 표 파싱, 예외 조건 처리까지 연결되므로 단계적으로 진행해야 합니다.

## 45. 이 프로젝트에서 가장 크게 배운 점은 무엇인가요?

가장 크게 배운 점은 AI 기능도 결국 안정적인 서비스 설계 위에 올라가야 한다는 것입니다. 모델을 붙이는 것보다 중요한 것은 어떤 질문을 검색으로 보낼지, 검색 근거가 맞는지, 실패했을 때 어떻게 fallback할지, 사용자의 피드백을 어떻게 다시 개선에 사용할지였습니다.

또한 새로운 기술을 빠르게 붙이는 것과 운영 가능한 형태로 녹이는 것은 다르다는 점을 배웠습니다. LangGraph, RAG, LoRA/QLoRA 같은 기술을 단순히 사용했다는 것보다, 어디에 쓰고 어디에는 쓰지 말아야 하는지 판단한 과정이 더 중요했습니다.

## 46. Spring MVC와 WebFlux 중 왜 MVC를 선택했나요?

이 프로젝트는 대량 스트리밍 처리보다 정형적인 요청-응답 API와 DB 저장 흐름이 중요합니다. 그래서 가장 보편적이고 설명하기 쉬운 Spring MVC를 선택했습니다.

WebFlux는 비동기 처리와 높은 동시성에 장점이 있지만, 처음 학습 프로젝트에서 사용하면 Controller, Service, Repository의 기본 책임을 이해하기 전에 복잡도가 커질 수 있습니다. 이번 프로젝트의 핵심은 Spring의 정석적인 계층 구조와 RAG 연동 흐름을 설명하는 것이므로 MVC가 더 적합하다고 판단했습니다.

## 47. 외부 Python RAG API 호출은 어떤 점을 조심해야 하나요?

가장 중요한 것은 HTTP 계약입니다. Spring과 Python이 서로 다른 런타임이기 때문에 요청 DTO, 응답 DTO, `Content-Type`, timeout, 실패 응답 형식을 명확히 맞춰야 합니다.

실제로 JSON body가 기대와 다르면 FastAPI에서 `422`가 발생할 수 있습니다. 그래서 Spring의 `RagClient`는 요청을 명확한 DTO로 만들고, 실패 시 fallback 경로로 넘어가도록 설계했습니다. 실무에서는 여기에 재시도 정책, circuit breaker, 호출 시간 제한을 추가할 수 있습니다.

## 48. 입력값 검증은 어떻게 해야 하나요?

사용자 질문은 비어 있거나 너무 길 수 있고, 의미 없는 문자열일 수도 있습니다. 따라서 Controller에서 기본적인 길이와 필수값을 검증하고, Service에서는 대화 의도와 금융 의도를 다시 판단해야 합니다.

이 프로젝트에서는 단순히 질문 문자열을 RAG에 바로 던지지 않고, 인사말, 종료 인사, 짧은 잡담, 상품 질문, 추천 질문을 나눕니다. 이렇게 한 이유는 검증이 단순한 null 체크가 아니라, 서비스 품질을 위한 입력 분류까지 포함하기 때문입니다.

## 49. 예외 처리는 어디에서 하는 것이 좋나요?

Controller는 HTTP 응답 형식을 맞추고, Service는 비즈니스 흐름의 실패를 처리하는 것이 좋습니다. Python RAG 호출 실패, DB 저장 실패, PDF 다운로드 파일 누락은 성격이 다르므로 각각 구분해야 합니다.

이 프로젝트에서는 RAG 호출 실패 시 바로 사용자에게 에러를 보여주기보다 fallback 답변을 사용합니다. 다만 실무에서는 모든 예외를 숨기면 운영자가 문제를 알 수 없기 때문에, 사용자 응답과 서버 로그를 분리해야 합니다. 사용자에게는 부드러운 안내를 보여주고, 서버에는 원인과 요청 id를 남기는 방식이 좋습니다.

## 50. CORS나 배포 환경 문제는 어떻게 봐야 하나요?

프론트는 Vercel 정적 사이트이고, API는 개인 Mac의 Spring Boot를 Cloudflare Tunnel로 노출합니다. 이 구조에서는 브라우저가 다른 origin으로 API를 호출하므로 CORS 설정이 필요할 수 있습니다.

문제 해결 접근은 단순합니다. 먼저 브라우저 개발자 도구에서 요청이 프론트 라우팅 문제인지, CORS 차단인지, API 서버 장애인지 나눕니다. 그 다음 Spring의 허용 origin과 Vercel의 API base URL, Cloudflare Tunnel 주소를 확인합니다. 배포 문제는 감으로 고치기보다 네트워크 요청을 기준으로 분리해야 합니다.

## 51. MySQL에서 검색 성능을 높이려면 무엇을 봐야 하나요?

현재 문서 수가 작기 때문에 단순 조회로도 동작합니다. 하지만 문서가 예금, 대출, 카드, 수수료까지 늘어나면 `product_name`, `category`, `source_file`, `page`, `created_at` 같은 컬럼에 인덱스를 고려해야 합니다.

또한 RAG 검색은 모든 chunk를 매번 애플리케이션 메모리에서 처리하면 한계가 옵니다. 실무 확장에서는 DB 인덱스, 전문 검색, vector DB 또는 MySQL vector 확장을 검토해야 합니다. 중요한 것은 처음부터 복잡한 저장소를 쓰는 것이 아니라, 데이터 양과 검색 병목이 실제로 생기는 시점에 근거를 가지고 바꾸는 것입니다.

## 52. PDF 재적재 시 데이터 정합성은 어떻게 지키나요?

문서 재적재는 단순 insert가 아닙니다. 같은 PDF를 여러 번 넣으면 중복 chunk가 생기고, 이전 버전 문서와 새 버전 문서가 섞여 답변될 수 있습니다.

그래서 재적재 시에는 source 파일명, 문서 해시, 버전, 적재 시간을 관리해야 합니다. 현재는 `--replace` 흐름으로 기존 chunk를 지우고 다시 넣는 단순한 방식을 사용할 수 있습니다. 실무에서는 문서 단위 트랜잭션을 잡고, 적재 중 실패하면 이전 정상 버전을 유지하는 방식이 더 안전합니다.

## 53. 답변 품질은 어떻게 평가하나요?

AI 답변 품질은 "그럴듯함"만 보면 안 됩니다. 이 프로젝트에서는 대표 질문을 만들어 인사, 추천, 가입대상, 상품 설명, 서류, 다운로드, 답변근거가 의도대로 동작하는지 확인합니다.

특히 금융 상품 추천은 조건 불일치가 치명적입니다. 예를 들어 10대에게 만 19세 이상 상품을 추천하면 품질 문제가 아니라 신뢰 문제입니다. 그래서 대표 질문 테스트에는 나이 조건, 군인 조건, 상품 설명 요청처럼 실패했을 때 바로 보이는 케이스를 포함했습니다.

## 54. RAG에서 hallucination은 어떻게 줄이나요?

첫째, 답변 생성 모델에 검색된 근거 밖의 내용을 만들지 말라고 지시해야 합니다. 둘째, 추천 로직은 가능하면 문서에서 추출한 조건을 기준으로 필터링해야 합니다. 셋째, 출처와 검색 근거를 사용자가 확인할 수 있어야 합니다.

이 프로젝트에서는 본문에는 자연스러운 답변만 보여주고, `답변근거` 버튼에 검색 score, snippet, PDF 다운로드를 숨겼습니다. 사용자는 대화 흐름을 방해받지 않고, 필요할 때 근거를 확인할 수 있습니다.

## 55. 금융상품 추천에서 법적·윤리적으로 조심할 점은 무엇인가요?

금융상품 추천은 사용자의 나이, 소득, 거주지, 군복무, 무주택 여부 같은 조건에 따라 달라질 수 있습니다. 따라서 단정적으로 "이 상품에 가입하세요"라고 말하면 위험합니다.

이 프로젝트의 답변은 상품 가입을 확정하는 것이 아니라, PDF 공시 기준으로 조건에 가까운 후보를 안내하는 수준이어야 합니다. 최종 가입 가능 여부는 은행 심사와 최신 공시를 확인해야 한다고 안내해야 합니다. 실무에서는 이 부분이 단순 문구가 아니라 서비스 책임 범위와 연결됩니다.

## 56. LangGraph를 에이전트 워크플로로 쓴 이유는 무엇인가요?

LangGraph는 복잡한 자율 에이전트를 만들기 위해서만 쓰는 도구가 아닙니다. 이 프로젝트에서는 문서 로드, 검색, 답변 생성 단계를 명확한 상태 흐름으로 표현하기 위해 사용했습니다.

처음에는 함수 호출만으로도 가능했습니다. 하지만 RAG 파이프라인은 단계별 입력과 출력, 실패 지점, 디버깅 포인트가 중요합니다. LangGraph로 나누면 어떤 단계에서 검색이 실패했는지, 생성이 실패했는지 설명하기 쉬워집니다.

## 57. LoRA/QLoRA 학습 데이터는 어떻게 검수하나요?

학습 데이터는 많이 모으는 것보다 정확하게 고르는 것이 중요합니다. 질문, 검색 근거, 좋은 답변, 나쁜 답변, 수정 답변을 함께 저장하고, 사람이 검수한 데이터만 학습 후보로 사용해야 합니다.

예를 들어 "나 군인인데"라는 질문에는 장병내일준비적금 안내가 자연스럽게 나와야 합니다. 이런 케이스를 성공/실패 데이터로 모으면 LoRA 학습에서 상담 말투와 도메인 응답 패턴을 개선할 수 있습니다. 반대로 근거가 틀린 답변을 그대로 학습하면 모델이 오류를 더 강하게 외우게 됩니다.

## 58. QLoRA를 실제로 하려면 어떤 환경이 필요한가요?

QLoRA는 양자화된 모델 위에서 adapter를 학습하므로 일반 LoRA보다 메모리 부담이 작습니다. 그래도 학습에는 GPU가 있는 환경이 유리합니다.

개인 Mac에서는 데이터셋 설계, 작은 샘플 검증, 추론 테스트를 하고, 본격 학습은 Hugging Face Jobs 같은 클라우드 GPU에서 진행하는 계획이 현실적입니다. 학습 결과는 adapter로 저장하고, 기존 RAG 검색 구조 위에 답변 생성 품질 개선용으로 붙입니다.

## 59. 이 프로젝트를 MSA로 확장한다면 어떤 순서로 나눌 건가요?

처음부터 21개 저장소 규모로 나누지는 않습니다. 먼저 경계가 분명한 기능부터 분리합니다.

예를 들어 문서 수집/적재, RAG 검색/생성, 사용자 질문 이력/피드백, 프론트 배포는 서로 변경 이유가 다릅니다. 따라서 1단계는 현재 모놀리식 Spring과 Python RAG를 안정화하고, 2단계는 문서 ingestion job을 분리하고, 3단계는 RAG inference API를 별도 서비스로 운영하는 순서가 자연스럽습니다. 트랜잭션은 질문 이력과 피드백 저장에 남기고, AI 응답 생성은 외부 호출로 분리합니다.

## 60. 면접에서 이 프로젝트의 핵심 경쟁력을 어떻게 말할 건가요?

이 프로젝트의 핵심은 "AI를 붙였다"가 아니라, AI 기능을 Spring 기반 서비스로 운영 가능하게 연결했다는 점입니다.

문서 적재, 검색 품질 개선, 자연스러운 답변, 근거 확인, PDF 다운로드, 모바일 시연, 배포, fallback, LoRA/QLoRA 학습 계획까지 하나의 흐름으로 이어집니다. 특히 질문이 엉뚱하게 들어왔을 때도 바로 검색하지 않고 의도를 판단하도록 개선한 과정은 실무적인 문제 발견과 해결 경험으로 설명할 수 있습니다.

## 61. 왜 product_knowledge.json 같은 상품 비교 지식표를 추가했나요?

처음에는 PDF chunk 검색만으로 추천과 비교 질문까지 처리하려고 했습니다. 하지만 `우대금리 제일 높은 상품`, `6개월 가능한 상품`, `30대 과장 추천`, `소액으로 시작할 적금`, `종류 있는대로 다` 같은 질문은 특정 문장 하나를 찾는 문제가 아니었습니다. 상품 전체를 비교하고 조건에 맞지 않는 후보를 제외해야 했습니다.

그래서 PDF를 다시 읽어 `product_knowledge.json`을 만들었습니다. 이 파일에는 상품명, 별칭, 태그, 가입대상, 가입기간, 가입금액, 기본금리, 우대금리, 중도해지, 예금자보호 같은 비교 필드가 들어갑니다. 챗봇은 추천·비교·기간·소액 질문에서는 이 구조화 데이터를 먼저 보고, 세부 근거 확인이 필요할 때 PDF chunk 검색을 사용합니다.

이 판단의 핵심은 금융상품 선택을 LLM 감에 맡기지 않는 것입니다. 상품 선택은 구조화 데이터와 규칙으로 하고, LLM은 그 결과를 자연스럽게 설명하는 역할로 제한하는 것이 더 안전하다고 봤습니다.

## 용어 메모

- CRUD: Create, Read, Update, Delete. 데이터를 생성, 조회, 수정, 삭제하는 기본 작업이다.
- RAG: 문서를 검색한 뒤 그 근거를 바탕으로 답변하는 방식이다.
- LangGraph: RAG나 에이전트 작업을 단계별 상태 그래프로 연결하는 도구다.
- fallback: 주 기능이 실패했을 때 대체 경로로 응답하는 방식이다.
- CORS: 브라우저가 다른 출처의 API를 호출할 때 적용하는 보안 정책이다.
- embedding: 문장을 의미 벡터로 바꾸는 방식이다.
- hybrid search: 키워드 검색과 의미 검색을 섞는 방식이다.
- citation: 답변의 근거가 되는 문서 정보다.
- product_knowledge.json: PDF에서 뽑은 상품별 가입대상, 기간, 금액, 금리, 우대조건을 담은 비교용 지식표다.
- hallucination: 모델이 근거 없는 내용을 그럴듯하게 만들어내는 현상이다.
- LoRA: 원본 모델은 고정하고 작은 adapter만 학습하는 방식이다.
- QLoRA: 양자화된 모델 위에서 LoRA adapter를 학습하는 방식이다.
- adapter: 원본 모델 옆에 붙는 작은 학습 모듈이다.
- OpenAI Responses API: OpenAI 모델에 입력을 보내고 답변을 받는 API 방식이다.
