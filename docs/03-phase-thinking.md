# 03. 차수별 사고 흐름

이 문서는 1차부터 현재까지 어떤 파일을 왜 바꿨는지 줄글로 정리한다.  
코드를 외우기보다 문제를 발견하고 해결해 온 흐름을 이해하는 것이 목적이다.

## 1차: Spring MVC와 기본 RAG 골격

처음 목표는 멋진 AI 서비스가 아니라 Spring MVC와 Python RAG가 MySQL을 중심으로 연결되는 가장 작은 구조를 만드는 것이었다. 그래서 Spring Boot에서 Controller, Service, Repository, Entity를 나누었다. Controller에 모든 코드를 넣지 않고 Service가 흐름을 관리하게 한 이유는 이후 Python 호출, DB 저장, 예외 처리, 피드백 저장이 붙을 것을 예상했기 때문이다.

`FinancialDocument`는 PDF chunk 지식 베이스로 만들고, `QueryHistory`는 사용자의 질문과 답변을 저장하기 위해 만들었다. 근거 목록은 처음부터 별도 테이블로 복잡하게 나누지 않고 JSON 문자열로 저장했다. 1차의 목적은 정규화된 분석 시스템이 아니라 질문하고 저장되는 흐름을 이해하는 것이었기 때문이다.

Python 쪽에는 FastAPI 서비스를 만들었다. Spring이 `/api/ask`를 받으면 Python `/ask`로 질문을 넘기고, Python은 MySQL 문서를 읽어 관련 문서를 찾았다. 처음 검색은 `TfidfVectorizer`였다. 한국어 조사 문제 때문에 단어 기준이 아니라 문자 n-gram으로 바꾸었고, 이 과정에서 RAG의 첫 문제는 모델보다 검색 단위라는 점을 배웠다.

Spring에서 Python을 호출할 때 `422` 오류도 있었다. 원인은 JSON body와 `Content-Type` 형식이었다. `RagClient`에서 DTO를 JSON 문자열로 바꾸고 `application/json`을 명시해 해결했다. 이후 Python이 꺼져도 화면 전체가 죽지 않도록 Spring Repository 기반 fallback도 추가했다. 1차의 결론은 RAG도 결국 웹 백엔드이고, 요청 흐름과 실패 대응이 먼저 안정되어야 한다는 것이었다.

## 2차: 부산은행 적립식예금 PDF로 범위 고정

1차는 일반 금융 문서 검색에 가까웠다. 하지만 학습과 시연에는 더 명확한 도메인이 필요했다. 그래서 범위를 `BNK부산은행 상품공시 > 예금상품 > 적립식예금`으로 고정했다. 처음부터 예금, 대출, 수수료를 모두 다루면 PDF 수집, 분류, 추출, 검색 품질 문제가 한꺼번에 커진다.

이 단계에서 `FinancialDocument`에 `institution`, `productName`, `productType`, `sourceUrl` 같은 메타데이터를 추가했다. 상품공시 RAG에서는 사용자가 보통 상품명 중심으로 묻기 때문이다. 예를 들어 `청년도약계좌 가입대상`은 제목보다 상품명 매칭이 더 중요하다.

원본 PDF와 처리 산출물도 나누었다. `data/raw`에는 원본 PDF를 두고, `data/processed`에는 다시 만들 수 있는 JSONL과 report를 둔다. PDF importer는 PyMuPDF로 페이지 텍스트를 뽑고 약 1,400자 단위로 chunk를 만든 뒤 MySQL에 저장한다. 상품명 추론에서는 짧은 상품명이 긴 상품명 안에 포함되는 문제가 있어 긴 alias부터 매칭하도록 바꾸었다.

프론트는 단순화했다. 사용자는 프론트를 학습하고 싶지 않다고 했고, 핵심은 Spring과 RAG였다. 그래서 문서 목록/등록 화면은 빼고, 빈 메인 화면과 우하단 플로팅 챗봇만 남겼다. 2차의 결론은 RAG 품질은 모델보다 데이터 구조와 범위 설정에서 먼저 결정된다는 것이었다.

## 3차: 로컬 LLM 답변 생성과 피드백 저장

2차까지는 검색 결과를 템플릿으로 묶어 보여주는 수준이었다. 3차에서는 검색된 근거를 바탕으로 실제 LLM 답변 생성을 시도했다. OpenAI API와 Hugging Face 로컬 모델 중 먼저 Hugging Face를 선택했다. 이유는 비용 없이 로컬에서 모델 실행 구조를 직접 이해할 수 있기 때문이다.

모델 후보는 Qwen과 Gemma를 비교했다. Qwen 소형 모델은 실행은 쉬웠지만 한국어 금융 문맥에서 답변 품질이 부족했다. Gemma 4 E4B full 모델은 다운로드는 가능했지만 Mac MPS에서 메모리 문제가 있었다. 그래서 Apple Silicon에 맞는 `mlx-community/gemma-4-e4b-it-4bit`와 MLX 실행 경로를 추가했다.

`rag-service/app.py`에는 `RAG_GENERATION_MODE`를 두었다. `mlx`는 로컬 Gemma, `template` 또는 실패 경로는 문서기반 fallback이다. 중요한 판단은 LLM이 실패해도 서비스가 멈추지 않게 하는 것이었다.

Spring 역량도 보강했다. 답변마다 `historyId`를 반환하고, 사용자가 `HELPFUL` 또는 `NOT_HELPFUL` 피드백을 남길 수 있게 했다. AI 서비스에서 피드백은 부가기능이 아니라 품질 개선의 출발점이다. 나중에 LoRA/QLoRA 학습 데이터 후보도 이 이력과 피드백에서 나온다.

## 4차: 검색 품질 개선

4차의 질문은 "답변은 그럴듯한데, 정말 맞는 근거를 찾고 있는가?"였다. 처음에는 상품명은 맞지만 같은 상품 안에서 엉뚱한 페이지가 먼저 뜨는 일이 있었다. 예를 들어 `펫 적금 우대금리 조건`에서 가입대상 페이지가 우대이율 페이지보다 앞설 수 있었다.

그래서 질문 의도를 나누었다. `가입대상`, `금리`, `가입기간`, `납입`, `해지`, `세제`, `서류`, `유의사항` 같은 항목을 감지하고, 문서에 강한 항목명이 있으면 점수를 더 주었다. 넓은 단어를 많이 잡으면 오히려 잡음이 늘어난다는 것도 확인했다. 그래서 강한 항목명과 약한 키워드를 나누어 재점수화했다.

그 뒤에도 자연어 표현 문제가 남았다. `혜택 받으려면 뭘 해야 해?`는 문서의 `우대이율`과 뜻은 가깝지만 단어가 다르다. 그래서 `intfloat/multilingual-e5-small` embedding을 붙였다. 최종 검색은 TF-IDF, embedding, 규칙 보정을 섞은 hybrid 방식이 되었다.

이 단계의 결론은 검색 품질은 단일 모델로 해결하는 것이 아니라 상품명 정확도, 질문 의도, 의미 유사도를 함께 설계해야 한다는 것이었다.

## 5차: Vercel과 Cloudflare Tunnel 배포

5차에서는 "내 컴퓨터에서는 되는데, 다른 사람이 어떻게 접속하지?"를 해결했다. 프론트는 HTML/CSS/JS 정적 파일이라 Vercel 무료 배포에 잘 맞는다. 반대로 Spring Boot, Python RAG, MySQL, 로컬 모델은 개인 Mac에서 실행해야 했다.

그래서 배포 구조를 `Vercel 정적 프론트 + Cloudflare Tunnel API`로 정했다. 사용자는 Vercel 화면에 접속하고, `/api/*` 요청은 `api.bnkaichat.xyz`를 통해 개인 Mac의 Spring Boot로 전달된다. 이렇게 하면 포트포워딩 없이 로컬 API를 외부에서 호출할 수 있다.

Vercel 404 문제도 있었다. 원인은 Vercel이 repo 루트를 기준으로 보는데 정적 파일은 `src/main/resources/static` 아래에 있었기 때문이다. 루트 `vercel.json`에 output directory를 명시해 해결했다. 커스텀 도메인은 `bnkaichat.xyz`와 `www.bnkaichat.xyz`를 Vercel 프론트로, `api.bnkaichat.xyz`는 Cloudflare Tunnel API로 역할을 분리했다.

5차의 결론은 배포도 아키텍처의 일부라는 것이다. 화면이 뜨는 것과 API 질문이 끝까지 답변되는 것은 별도로 확인해야 한다.

## 6차: 질문 라우팅과 추천 조건 검증

6차의 출발점은 `ㅎㅇ`, `예금 종류`, `50대 추천`이 엉뚱한 답변으로 흐르는 문제였다. 모든 입력을 RAG 검색으로 보내면 인사말도 상품 문서와 매칭되고, 예금 분류 질문도 약관 문장으로 갈 수 있다.

그래서 검색 전 직접 응답 라우팅을 추가했다. 인사, 감사, 종료 표현은 문서 검색으로 보내지 않고 자연스럽게 응답한다. `예금 종류`는 예금상품 분류를 직접 안내한다. 반대로 상품명이나 금융 의도가 있으면 RAG 검색으로 보낸다.

추천 흐름도 분리했다. 추천은 단순 검색이 아니라 조건에 맞지 않는 상품을 제외하는 문제다. `10대인데 추천해주세요`에서 청년도약계좌가 추천되는 문제가 발견되면서, 상품명에 `청년`이 있는지만 볼 것이 아니라 PDF의 `가입대상`, `가입자격`, `가입제한` 구간을 읽어야 한다는 판단을 했다. 그래서 `만 19세 이상`, `만 34세 이하`, `연령에 관계없이`, `제한없음` 같은 조건을 추출해 나이 조건과 비교하도록 바꾸었다.

6차의 결론은 좋은 RAG는 무조건 검색을 많이 하는 것이 아니라, 검색이 필요한 질문과 그렇지 않은 질문을 먼저 나누는 것이라는 점이었다.

## 7차: 답변근거와 학습 데이터 설계

7차의 질문은 "답변은 나오는데, 왜 그 답변이 나왔는지 확인할 수 있는가?"였다. 금융 상품공시처럼 근거가 중요한 분야에서는 어떤 PDF chunk가 선택됐고 점수가 얼마인지 볼 수 있어야 한다.

그래서 `GET /api/histories/{historyId}/evidence` API를 추가했다. 새 검색을 다시 돌리지 않고, 답변 당시 `query_histories`에 저장한 `citationsJson`을 읽어 순위, 제목, score, snippet을 보여준다. 프론트에서는 이 기능을 크게 드러내지 않고 작은 `답변근거` 버튼으로 제공한다.

같은 흐름에서 LoRA/QLoRA 데이터셋도 설계했다. 중요한 판단은 상품 지식을 모델에 외우게 하지 않는 것이다. 금리, 가입대상, 서류는 바뀔 수 있으므로 최신 사실은 RAG가 찾아야 한다. LoRA/QLoRA는 검색된 근거를 읽고 자연스럽고 안전하게 답하는 습관을 학습시키는 용도로 잡았다.

## 8차: GPT처럼 자연스러운 답변

8차의 목표는 답변이 양식처럼 보이지 않게 만드는 것이었다. 지금까지의 개선은 검색 정확도와 근거 확인에 집중했다. 그러나 실제 챗봇 시연에서는 사용자가 먼저 보는 것은 검색 score가 아니라 말투다.

중요한 판단은 특정 3개 케이스를 땜질하지 않는 것이었다. 입력이 무엇이든 먼저 의도를 이해하고 자연스럽게 대답해야 한다. `나 5세임`이면 어린이라 직접 가입은 어렵고 보호자 확인이 필요하다고 자연스럽게 말해야 한다. `나 군인인데`라면 국군장병을 응원한다는 짧은 반응 뒤 장병내일준비적금을 안내하는 것이 자연스럽다. `상품에 대해 설명해줘`라면 다운로드만 주는 것이 아니라 본문에서 내용을 요약해야 한다.

기술적으로는 `rag-service/app.py`의 질문 라우팅, 프롬프트, fallback 답변을 다시 잡았다. 본문에서는 `핵심 답변`, `확인 내용`, `출처` 같은 제목을 줄이고, 근거와 다운로드는 답변근거 버튼에 숨긴다. 로컬 Gemma만으로 자연스러운 한국어가 부족한 상황에 대비해 OpenAI Responses API를 호출하는 `RAG_GENERATION_MODE=openai` 선택 모드도 추가했다.

8차의 목표는 한 문장으로 정리된다. **기본은 GPT처럼 자연스럽게 대화하고, 금융 정보가 필요한 순간에는 RAG 근거로 정확하게 답한다.**

현재 1차 개선으로 `설명/알려줘/뭐야` 같은 상품 설명 의도를 추가했고, `s%`, `ㅋㅋ`, `잘가` 같은 입력은 문서 검색으로 보내지 않도록 정리했다. 또 로컬 Gemma가 생성한 답변에서 깨진 bullet이나 끝나지 않은 문장을 후처리해 모바일 챗봇에서 보기 어색하지 않게 했다. 대표 질문은 18개로 늘렸고 모두 통과했다.

## 용어 메모

- Controller: HTTP 요청을 받는 Spring 계층이다.
- Service: 비즈니스 흐름과 트랜잭션을 담당하는 Spring 계층이다.
- Repository: DB 접근을 담당하는 계층이다.
- Entity: DB 테이블과 매핑되는 Java 객체다.
- TF-IDF: 단어 중요도를 계산해 문서를 검색하는 방식이다.
- n-gram: 글자를 일정 길이로 잘라 검색 단위로 쓰는 방식이다.
- embedding: 텍스트를 의미 벡터로 바꾸는 방식이다.
- hybrid search: 키워드 검색과 의미 검색을 섞는 방식이다.
- fallback: 주 경로가 실패했을 때 대체 경로로 응답하는 방식이다.
- 질문 라우팅: 입력을 검색 전 어떤 처리 경로로 보낼지 나누는 과정이다.
- 회귀 질문: 기능 수정 후에도 핵심 질문이 계속 잘 답변되는지 확인하는 대표 질문이다.
- 답변근거: 검색 score, snippet, PDF 다운로드를 답변 아래 작은 버튼으로 확인하는 기능이다.
- LoRA: 작은 adapter만 학습해 모델의 답변 습관을 바꾸는 방식이다.
- QLoRA: 양자화된 모델 위에서 LoRA 학습을 수행하는 방식이다.
- GPT API: OpenAI 모델을 API로 호출해 답변 생성을 맡기는 방식이다.
