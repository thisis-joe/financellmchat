# 05. LoRA와 QLoRA 학습 계획

이 문서는 LoRA/QLoRA를 언제, 어디서, 어떻게, 왜 사용할지 정리한다.  
핵심은 상품 지식을 모델에 외우게 하는 것이 아니라, RAG 근거를 읽고 자연스럽게 답하는 습관을 학습시키는 것이다.

## 먼저 결론

LoRA/QLoRA는 지금 프로젝트에서 반드시 해야 할 학습 주제다. 다만 역할을 잘못 잡으면 위험하다.

좋은 역할:

- 검색된 근거를 읽고 자연스러운 상담 답변으로 바꾸는 법 학습
- 짧고 애매한 질문에 먼저 자연스럽게 반응하는 말투 학습
- 근거에 없는 내용은 단정하지 않는 습관 학습
- 상품 추천 시 조건이 맞지 않는 후보를 조심스럽게 제외하는 답변 방식 학습
- 출처와 다운로드를 본문에 늘어놓지 않고 답변근거로 분리하는 응답 정책 학습

나쁜 역할:

- 특정 상품의 금리, 가입대상, 제출서류를 모델에 외우게 하기
- 최신 공시 내용을 모델 파라미터에 저장하기
- RAG 검색 없이 모델 기억만으로 답하게 만들기

금융 상품 조건은 바뀔 수 있다. 따라서 최신 사실은 항상 PDF 검색으로 가져오고, LoRA/QLoRA는 답변 태도와 형식을 보정하는 데 쓴다.

## 왜 지금 바로 학습하지 않는가

LoRA/QLoRA는 데이터가 있어야 의미가 있다. 현재는 먼저 RAG 검색과 답변 흐름을 안정화하는 단계다. 검색이 틀린 상태에서 학습 데이터를 만들면 모델은 틀린 답변 습관을 배운다.

그래서 순서는 다음이 맞다.

```text
1. 대표 질문과 검색근거 안정화
2. 상품 비교 지식표로 추천·비교 질문 안정화
3. 자연스러운 답변 생성 목표 확정
4. query_histories와 answer_feedbacks 수집
5. 좋은 답변과 나쁜 답변 분류
6. 사람이 고친 정답 답변 작성
7. LoRA용 JSONL 데이터셋 생성
8. 작은 LoRA 실험
9. 필요하면 QLoRA로 큰 모델 실험
```

즉, 학습은 뒤로 미루는 것이 아니라, 학습할 만한 데이터를 만들기 위한 순서를 밟는 것이다.

## LoRA와 QLoRA 차이

LoRA는 원본 모델 전체를 다시 학습하지 않고 작은 adapter만 학습한다. 학습 비용이 낮고, 원본 모델은 그대로 둔 채 프로젝트에 맞는 말투와 응답 방식을 얹을 수 있다.

QLoRA는 양자화된 모델 위에서 LoRA adapter를 학습한다. 더 큰 모델을 적은 GPU 메모리로 실험할 때 유리하다. 데이터 포맷은 거의 같고, 차이는 학습 런타임과 메모리 전략이다.

```text
LoRA
  -> 원본 모델 고정
  -> 작은 adapter만 학습
  -> 먼저 시도하기 좋음

QLoRA
  -> 원본 모델을 4bit 등으로 양자화
  -> adapter만 학습
  -> 큰 모델을 제한된 메모리에서 실험할 때 사용
```

설명할 때는 이렇게 말하면 된다.

> 이 프로젝트에서 LoRA/QLoRA는 상품 정보를 외우기 위한 학습이 아닙니다. RAG가 찾은 근거를 읽고, 사용자 의도에 맞게 자연스럽고 안전하게 답하는 방식을 학습시키기 위한 실험입니다.

## 어떤 데이터를 쓸 것인가

출발점은 Spring이 저장한 이력이다.

```text
query_histories
  -> 질문
  -> 당시 답변
  -> citationsJson 검색 근거
  -> 상태

answer_feedbacks
  -> HELPFUL / NOT_HELPFUL
  -> 사용자 평가
  -> 수정할 답변 후보
```

`HELPFUL` 답변은 좋은 후보가 될 수 있다. `NOT_HELPFUL` 답변은 그대로 학습시키지 않고 사람이 고쳐야 한다. 예를 들어 10대에게 청년도약계좌를 추천한 답변은 나쁜 데이터이며, 고친 답변은 `만 19세 이상 조건이 있어 제외`하는 방향이어야 한다.

## 권장 JSONL 포맷

파일 예시:

```text
data/training/lora/rag_answer_sft.jsonl
```

한 줄은 하나의 학습 예시다.

```json
{
  "id": "bnk-rag-sft-000001",
  "task": "rag_natural_answer",
  "messages": [
    {
      "role": "system",
      "content": "당신은 BNK부산은행 상품공시 PDF를 근거로 답하는 상담 보조 챗봇입니다. 근거에 없는 내용은 추측하지 않고, 사용자의 말투와 상황에 맞게 자연스럽게 답합니다."
    },
    {
      "role": "user",
      "content": "질문: 나 군인인데\n\n검색 근거:\n[1] 문서: 부산은행 장병내일준비적금 상품설명서\n상품명: 부산은행 장병내일준비적금\n내용: 군 복무자를 대상으로 하는 적금 상품 안내..."
    },
    {
      "role": "assistant",
      "content": "부산은행은 국군장병을 응원합니다.\n\n군 복무 중이라면 먼저 볼 상품은 부산은행 장병내일준비적금입니다. 가입자격, 제출서류, 만기 수령 조건은 복무 구분에 따라 달라질 수 있으니 답변근거에서 상품설명서를 확인해 주세요."
    }
  ],
  "metadata": {
    "domain": "BNK부산은행 상품공시",
    "category": "예금상품>적립식예금",
    "source_history_id": 123,
    "citations": [
      {
        "documentId": 1,
        "title": "부산은행 장병내일준비적금 상품설명서",
        "productName": "부산은행 장병내일준비적금",

    "split": "train"
  }

```

모델 학습에 핵심으로 쓰이는 것은 `messages`다. `metadata`는 추적과 평가를 위한 정보다.

## 학습 데이터 품질 기준

좋은 데이터:

- 첫 문장이 자연스럽다.
- 사용자의 상황을 먼저 이해한다.
- 상품 정보는 검색 근거 안에서만 말한다.
- 추천할 때 조건이 맞지 않는 상품은 제외한다.
- PDF 다운로드나 출처 목록을 본문에 길게 늘어놓지 않는다.
- 필요하면 답변근거 확인을 짧게 안내한다.

나쁜 데이터:

- `핵심 답변`, `확인 내용`, `출처` 같은 양식 제목이 본문에 반복된다.
- 근거와 다른 상품을 추천한다.
- 청년 전용 상품을 50대에게 추천한다.
- 만 19세 이상 상품을 10대에게 추천한다.
- 근거가 부족한데 단정한다.
- 모델이 프롬프트나 문서 제목만 반복한다.

## 학습 예시 유형

처음에는 20~30개 정도를 사람이 검수해 만든다.

포함할 유형:

- 인사말: `ㅎㅇ`, `안녕`
- 종료/감사: `ㅂㅇ`, `수고`, `고마워`
- 무의미 입력: `s%`, `ㅋㅋ`
- 나이 조건: `10대인데 추천`, `만 18세`, `50대`, `90세`
- 상황 조건: `나 군인인데`, `반려동물 키워`
- 상품 설명: `펫 적금 설명해줘`, `청년도약계좌 알려줘`
- 가입대상: `청년도약계좌 가입대상`
- 금리/혜택: `펫 적금 혜택 받으려면`
- 서류/해지: `장병내일준비적금 만기 서류`
- 범위 밖 질문: `대출도 알려줘`, `카드 혜택은?`
- 비교 질문: `우대금리 제일 높은 게 뭐야`, `6개월 가능한 상품 있어?`, `소액으로 시작할 적금 추천해줘`

## 평가 방법

학습 후에도 평가는 RAG답게 해야 한다.

좋은 결과:

- 답변이 자연스럽다.
- 근거에 없는 내용을 덜 말한다.
- 조건 불일치 상품을 추천하지 않는다.
- 대표 질문에서 답변이 덜 흔들린다.
- 사용자는 본문을 읽고 이해하고, 필요할 때 답변근거를 확인할 수 있다.

나쁜 결과:

- 모델이 상품 조건을 외운 것처럼 말한다.
- 검색 근거가 부족해도 단정한다.
- 출처나 파일명을 꾸며낸다.
- 현재 적재되지 않은 대출/카드 자료까지 아는 척한다.

## GPT API와의 관계

GPT API는 LoRA/QLoRA와 경쟁 관계가 아니다. GPT API는 지금 당장 자연스러운 답변 품질을 올리는 방법이고, LoRA/QLoRA는 나중에 로컬 모델도 프로젝트 답변 정책을 잘 따르게 만드는 학습 실험이다.

현실적인 진행은 다음과 같다.

```text
GPT API 또는 개선된 로컬 모델로 좋은 답변 기준 만들기
  -> 피드백과 답변근거 수집
  -> 사람이 좋은 학습 예시로 정리
  -> LoRA 학습
  -> 필요하면 QLoRA로 확장
```

현재 코드는 `RAG_GENERATION_MODE=openai`를 선택하면 OpenAI Responses API를 통해 답변을 생성할 수 있게 준비되어 있다. 이 경로로 좋은 답변 예시를 모으면, 나중에 LoRA/QLoRA 학습 데이터의 정답 답변 초안으로도 활용할 수 있다. 단, 최종 학습 데이터는 사람이 근거 일치 여부를 검수한 뒤 넣어야 한다.

## 용어 메모

- SFT: 질문과 정답 답변 예시로 모델을 지도학습하는 방식이다.
- LoRA: 원본 모델은 고정하고 작은 adapter만 학습하는 파인튜닝 방식이다.
- QLoRA: 양자화된 모델 위에서 LoRA adapter를 학습하는 방식이다.
- adapter: 원본 모델 옆에 붙는 작은 학습 모듈이다.
- JSONL: 한 줄에 JSON 객체 하나를 저장하는 데이터 파일 형식이다.
- messages 포맷: system, user, assistant 역할을 가진 대화형 학습 데이터 구조다.
- metadata: 학습에는 직접 쓰지 않아도 추적과 평가를 위해 함께 저장하는 부가 정보다.
- train/validation/test: 학습용, 검증용, 최종 평가용 데이터 분할이다.
- 근거 일치율: 답변 내용이 검색된 문서 근거와 얼마나 일치하는지 보는 기준이다.
- 상품 비교 지식표: 추천과 비교를 위해 PDF에서 추출한 금리, 기간, 금액, 대상 조건을 구조화한 데이터다.
- 양자화: 모델 가중치 정밀도를 낮춰 메모리 사용량을 줄이는 방식이다.
