# 11. 7차 사고 흐름

7차의 출발점은 "답변은 나오는데, 왜 그 답변이 나왔는지 사용자가 확인할 수 있는가?"였다.  
RAG 프로젝트에서 답변 자체만 보여주면 설명력이 부족하다. 특히 금융 상품공시처럼 근거가 중요한 분야에서는 어떤 PDF 조각이 선택됐고, 그 조각이 답변에 어떤 영향을 줬는지 확인할 수 있어야 한다.

이미 답변 하단에는 출처 카드가 있었다. 하지만 출처 카드는 사용자가 PDF를 다운로드하는 용도에 가깝고, 검색 품질을 확인하기에는 부족했다. 검색 점수, snippet, 선택 순위가 보여야 "검색이 맞았는지"를 볼 수 있다. 그래서 새 DB 테이블을 만들기보다, 이미 `query_histories`에 저장하던 `citationsJson`을 재사용하기로 했다.

Spring에는 `GET /api/histories/{historyId}/evidence` API를 추가했다. 사용자가 질문하면 Spring은 기존처럼 질문 이력을 저장하고 `historyId`를 반환한다. 프론트는 이 `historyId`를 들고 있다가 사용자가 `답변근거` 버튼을 누르면 근거 API를 호출한다. 응답에는 질문, 상태, 요약, 근거 문서 목록, 검색 점수, snippet이 담긴다.

프론트에서는 이 기능을 크게 보이게 만들지 않았다. 사용자의 기본 흐름은 질문과 답변을 보는 것이고, 근거 디버그는 필요할 때만 펼치면 된다. 그래서 답변 아래에 아주 작은 `답변근거` 버튼을 두었다. 버튼을 누르면 작은 패널이 열리고, 다시 누르면 닫힌다. 직접 응답처럼 검색을 하지 않은 답변은 "문서 검색이 필요하지 않았다"는 설명을 보여준다.

로그도 함께 남겼다. `RagService`는 질문 이력을 저장한 뒤 historyId, 상태, 상위 근거 제목과 점수를 로그에 남긴다. 운영 로그를 보면 특정 답변이 어떤 문서를 근거로 했는지 빠르게 추적할 수 있다. 이것은 멋진 기능이라기보다 RAG 서비스를 운영 가능한 백엔드로 만들기 위한 기본 장치다.

두 번째 작업은 LoRA/QLoRA 데이터셋 설계였다. 여기서 가장 중요한 판단은 "상품 지식을 모델에 외우게 하지 않는다"는 것이다. 예금 금리, 가입대상, 제출서류는 바뀔 수 있다. 이 정보를 모델 파라미터에 넣으면 공시가 바뀐 뒤에도 예전 정보를 자신 있게 말할 수 있다. 그래서 최신 사실은 RAG 검색으로 가져오고, LoRA/QLoRA는 근거를 읽고 안정적으로 답하는 형식만 학습시키는 용도로 잡았다.

학습 데이터 포맷은 chat messages 기반 JSONL로 정리했다. `system`에는 역할과 안전 규칙을 넣고, `user`에는 질문과 검색 근거를 함께 넣고, `assistant`에는 기대 답변을 넣는다. `metadata`에는 historyId, 출처 문서, 상품명, score, feedback, split을 남긴다. 모델 학습에는 messages가 핵심이고, metadata는 추적과 평가를 위한 정보다.

LoRA와 QLoRA의 데이터 포맷은 거의 같다. 차이는 학습 방식이다. LoRA는 원본 모델 위에 작은 adapter를 학습하고, QLoRA는 양자화된 모델 위에서 adapter를 학습해 메모리를 줄인다. 그래서 현재 단계에서는 먼저 데이터 품질과 평가 기준을 잡고, 작은 LoRA 실험으로 답변 형식 개선이 보이는지 확인한 뒤, 더 큰 모델이나 메모리 문제가 생기면 QLoRA로 확장하는 순서가 맞다.

7차의 결론은 "설명 가능한 RAG와 학습 가능한 RAG는 같은 방향을 본다"는 것이다. 답변근거 API는 지금 답변의 근거를 보여주고, LoRA 데이터셋 포맷은 좋은 근거 기반 답변을 나중에 학습 데이터로 바꿀 수 있게 한다. 즉, 운영 로그, 사용자 피드백, 검색 근거, 학습 데이터가 하나의 흐름으로 이어진다.

## 다음 과제

다음 단계에서는 대표 질문을 20~30개로 늘리는 것이 좋다. 질문이 늘어나야 `HELPFUL` 답변과 `NOT_HELPFUL` 답변이 쌓이고, LoRA 데이터 후보를 고를 수 있다. 또한 `답변근거` 패널에서 실제로 엉뚱한 문서가 보이는 사례를 모으면 검색 로직 개선의 기준이 된다.

LoRA/QLoRA는 바로 학습을 돌리기보다 먼저 샘플 JSONL 20개 정도를 사람이 검수해서 만드는 편이 좋다. 모델 학습보다 데이터 품질이 먼저다.

## 용어 메모

- 답변근거 API: 저장된 질문 이력의 citationsJson을 읽어 검색 근거 문서, score, snippet을 돌려주는 API다.
- snippet: 검색된 문서 chunk에서 질문과 관련 있는 짧은 발췌문이다.
- score: 검색 결과가 질문과 얼마나 관련 있다고 판단했는지 나타내는 점수다.
- historyId: 한 번의 질문과 답변 이력을 식별하는 ID다.
- SFT: 질문과 정답 답변 예시로 모델을 지도학습하는 방식이다.
- LoRA: 원본 모델은 고정하고 작은 adapter만 학습하는 방식이다.
- QLoRA: 양자화된 모델 위에서 LoRA adapter를 학습해 메모리 사용을 줄이는 방식이다.
- metadata: 학습에는 직접 쓰지 않더라도 추적, 필터링, 평가를 위해 함께 저장하는 부가 정보다.
