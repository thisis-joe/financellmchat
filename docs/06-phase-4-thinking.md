# 06. 4차 사고 흐름

4차의 출발점은 "답변이 자연스럽게 나오는데, 정말 질문에 맞는 근거를 찾고 있는가?"라는 의심이었다.  
LLM을 붙이기 전에도 답변은 그럴듯해 보일 수 있다. 하지만 RAG에서는 답변보다 먼저 검색 근거가 맞아야 한다. 그래서 검색 결과를 직접 확인했다.

처음 문제는 상품명은 맞지만 같은 상품 안에서 엉뚱한 페이지가 먼저 뜨는 것이었다. 예를 들어 `펫 적금 우대금리 조건은?`이라고 물으면 `펫 적금` 상품은 찾지만, 우대이율 표가 있는 페이지 대신 가입대상이나 기본이율 페이지가 섞일 수 있었다. 기존 TF-IDF는 같은 단어가 많이 겹치는 문서를 잘 찾지만, 질문이 원하는 항목을 이해하지는 못했다.

그래서 먼저 규칙 기반 재점수화를 넣었다. `rag-service/app.py`에서 질문을 분석해 상품명과 의도를 감지했다. 의도는 `가입대상`, `금리`, `가입기간`, `납입`, `해지`, `세제`, `서류`, `유의사항`으로 나누었다. 그리고 문서에 `가입자격`, `우대이율`, `만기해지`, `신청서류` 같은 강한 항목명이 있으면 점수를 더 주었다.

처음에는 `조건` 같은 단어를 넓게 잡았다. 그런데 `우대금리 조건`이라는 질문에서 `가입대상` 페이지가 올라오는 문제가 생겼다. 이때 얻은 교훈은 "키워드를 많이 잡는 것이 항상 좋은 검색은 아니다"라는 점이었다. 넓은 단어는 잡음도 함께 키운다. 그래서 강한 항목명과 약한 키워드를 나누고, 질문 맥락에 맞게 점수를 다르게 주었다.

snippet도 고쳤다. 이전에는 문서 앞부분을 잘라 보여주었다. 하지만 사용자는 근거가 어디인지 보고 싶어 한다. 그래서 질문 의도와 가까운 단어 주변을 잘라 snippet으로 만들었다. 이 변경으로 검색 결과가 왜 선택되었는지 더 잘 보이게 되었다.

규칙 기반 보정 뒤에도 한계는 남았다. 사용자가 `혜택 받으려면 뭘 해야 해?`, `누가 만들 수 있어?`, `만기 때 어떤 서류가 필요해?`처럼 자연어로 묻는 경우다. 이런 질문은 `우대이율`, `가입자격`, `만기해지`와 뜻은 가깝지만 단어가 다르다. 이 문제를 해결하기 위해 embedding 검색을 붙였다.

embedding은 문장을 숫자 벡터로 바꾸고 의미가 가까운 문서를 찾는다. 하지만 embedding만 단독으로 쓰지는 않았다. 금융 상품 검색에서는 정확한 상품명과 항목명이 중요하기 때문이다. 그래서 최종 점수는 `TF-IDF 45% + embedding 55% + 상품명/항목명 규칙 보정`으로 만들었다.

기본 embedding 모델은 `intfloat/multilingual-e5-small`로 정했다. 한국어를 포함한 다국어 검색에 사용할 수 있고, E5 계열은 `query:`와 `passage:` 접두어를 쓰는 검색 방식이 명확하다. 구현은 MySQL에 embedding을 저장하지 않고 Python 메모리에서 계산해 캐시하는 방식으로 시작했다. 현재 문서 수가 196개 수준이라 이 방식이 단순하고 충분하다고 판단했다.

테스트 중 dtype 경고도 만났다. embedding 행렬과 질문 벡터의 내적을 계산할 때 `matmul` 경고가 발생했다. 값은 나오지만 경고를 남겨두면 나중에 진짜 수치 문제와 구분하기 어렵다. 그래서 벡터를 `float32`로 고정하고 `np.dot`을 사용하도록 바꾸었다.

검증 결과 `펫 적금 혜택 받으려면 뭘 해야 해?`는 우대이율 페이지를, `장병내일준비적금 만기 때 어떤 서류가 필요해?`는 만기해지와 제출서류 페이지를 1순위로 찾았다. 이 단계의 결론은 "검색은 한 번에 모델로 해결하는 것이 아니라, 상품명 정확도와 의미 유사도를 함께 설계해야 한다"는 것이다.

그 다음 문제는 답변 모양이었다. 검색은 맞더라도 사용자에게 문서 제목과 점수만 보여주면 "그래서 뭘 하면 되는데?"라는 질문이 남는다. 그래서 LLM 프롬프트와 fallback 답변을 모두 `핵심 답변`, `확인 내용`, `출처`, `추가 확인 필요` 구조로 맞추었다. 예를 들어 `펫 적금 혜택 받으려면 뭘 해야 해?`라고 물으면 우대이율 조건을 보여주고, `장병내일준비적금 만기 때 어떤 서류가 필요해?`라고 물으면 확인서와 제출서류를 보여준다.

출처 표기도 정리했다. DB 안의 chunk 제목에는 `p2-1` 같은 내부 구분이 붙어 있을 수 있지만, 사용자가 볼 때는 너무 자세하고 산만하다. 그래서 최종 답변과 화면의 출처 카드는 원본 파일 제목만 남기도록 했다.

4차에서 수정한 핵심 파일은 `rag-service/app.py`와 `rag-service/requirements.txt`다. 문서도 함께 정리했다. 앞으로 "문서정리"라는 요청이 오면 docs 전체를 읽고, 중복을 합치고, 번호 순서대로 자연스럽게 다시 구성하는 작업으로 처리하기로 했다.

## 다음 과제

현재 embedding은 메모리 캐시 방식이다. 문서가 수천 개 이상으로 늘어나면 MySQL에 embedding을 저장하거나 FAISS 같은 vector index를 검토해야 한다. 또한 검색 결과가 정말 좋아졌는지 감으로만 볼 것이 아니라, 질문과 기대 근거 문서를 묶은 평가셋을 만들어야 한다.

## 용어 메모

- 재점수화: 1차 검색 점수에 다른 기준을 더해 최종 순위를 다시 매기는 방식이다.
- 질문 의도: 사용자가 알고 싶은 정보의 종류다. 예를 들어 가입대상, 금리, 해지, 서류 등이 있다.
- snippet: 검색 결과에서 보여주는 짧은 근거 문장이다.
- embedding: 텍스트를 의미를 담은 숫자 벡터로 바꾸는 방식이다.
- vector index: 많은 벡터 중 가까운 벡터를 빠르게 찾기 위한 검색 구조다.
- FAISS: Meta가 만든 벡터 검색 라이브러리다.
- dtype: 숫자 배열이 어떤 정밀도와 형식으로 저장되는지 나타내는 타입이다.
- 내부 chunk 번호: 하나의 PDF를 검색하기 위해 나눈 조각의 번호다. 디버깅에는 필요하지만 사용자 답변에는 보통 숨긴다.
