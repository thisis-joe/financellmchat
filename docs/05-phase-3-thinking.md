# 05. 3차 사고 흐름

3차의 목표는 검색된 문서를 바탕으로 실제 LLM 답변 생성을 시도하는 것이었다.  
2차까지는 검색 결과를 템플릿 문장으로 묶어 보여주었다. 이 방식은 안정적이지만 사용자가 묻는 가입 조건, 우대금리, 유의사항을 자연스럽게 요약하지는 못했다.

처음 선택지는 OpenAI API와 Hugging Face 로컬 모델이었다. OpenAI API는 품질과 속도가 좋지만 API 키와 비용 관리가 필요하다. 반면 Hugging Face 모델은 로컬에서 직접 실행할 수 있어 학습용으로 구조를 이해하기 좋다. 그래서 3차에서는 Hugging Face를 먼저 붙이고, 필요하면 나중에 API 방식을 비교하기로 했다.

모델 후보는 Qwen 계열과 Gemma 계열을 비교했다. 가벼운 Qwen 모델은 실행은 쉬웠지만 한국어 금융 PDF 문맥에서 표현이 깨지거나 답변 품질이 부족했다. 그래서 기준 모델은 Gemma 4 E4B로 잡았다. 다만 full `google/gemma-4-E4B-it`는 로컬 Mac MPS에서 메모리 문제가 있었다. 다운로드는 가능했지만 실행 중 `Invalid buffer size` 문제가 발생했다.

이 문제를 모델 선택 실패로 보지는 않았다. 같은 모델이라도 실행 런타임과 양자화 여부에 따라 로컬 실행 가능성이 달라진다. 그래서 Apple Silicon에 맞는 `mlx-community/gemma-4-e4b-it-4bit`를 `mlx-vlm`으로 실행하는 경로를 추가했다. 이 방식은 약 5.3GB peak memory로 실제 답변 생성에 성공했다.

`rag-service/app.py`에는 `RAG_GENERATION_MODE`를 두었다. `hf`는 Transformers 경로, `mlx`는 MLX 경로다. 현재 시연용 실행은 `mlx`를 사용해 답변 생성에도 모델을 활용한다. 중요한 판단은 LLM을 붙이더라도 실패하면 문서기반 추출 답변으로 돌아가게 한 것이다. 모델 다운로드, 메모리, 추론 실패 때문에 전체 서비스가 멈추면 안 되기 때문이다.

프롬프트도 단순히 질문만 넣지 않았다. 검색된 chunk를 근거로 넣고, 근거에 없는 내용은 추측하지 말라고 지시했다. 답변 형식은 `핵심 답변`, `확인 내용`, `출처`, `추가 확인 필요`로 제한했다. 출처는 내부 chunk 번호가 아니라 파일 제목만 쓰도록 했다. 처음에는 chat template을 제대로 적용하지 않아 모델이 질문을 반복하거나 지시문 일부를 복사하는 문제가 있었다. 그래서 `apply_chat_template`를 명시적으로 적용했다.

이 시점에서 중요한 사실도 확인했다. LLM을 붙였다고 RAG 품질이 자동으로 좋아지지는 않는다. 검색된 chunk가 부족하면 모델도 부족한 답변을 만든다. 그래서 3차의 결론은 "LLM 답변 생성은 가능하지만, 다음 핵심은 검색 품질 개선"이었다.

3차 중간에 Spring 역량도 보강했다. 프로젝트가 Python RAG 호출 래퍼처럼 보이면 Spring 백엔드 역량이 약해 보일 수 있었다. 그래서 답변마다 `historyId`를 반환하고, 사용자가 `HELPFUL` 또는 `NOT_HELPFUL` 피드백을 남길 수 있게 했다. 이를 위해 `AnswerFeedback`, `FeedbackService`, `FeedbackApiController`, 전역 예외 처리, MockMvc 테스트를 추가했다.

이 기능은 단순 부가기능이 아니다. AI 서비스에서는 답변 품질을 다시 시스템으로 회수하는 구조가 중요하다. 나중에 `NOT_HELPFUL` 답변을 모아 검색 품질을 개선하거나, 좋은 답변을 LoRA 학습 데이터 후보로 삼을 수 있기 때문이다. Spring은 여기서 요청 검증, 트랜잭션, JPA 연관관계, 예외 응답을 담당한다.

3차에서 수정한 핵심 파일은 `rag-service/app.py`, `requirements-llm.txt`, Spring의 `rag`, `feedback`, `common` 패키지, 그리고 테스트 파일이었다. 이 단계의 결론은 "LLM은 답변을 자연스럽게 만들 수 있지만, 운영 흐름과 실패 대응은 Spring에서 잡아야 한다"는 것이었다.

## LoRA와 QLoRA를 아직 실행하지 않은 이유

LoRA와 QLoRA는 바로 실행하지 않았다. 금융 상품 조건처럼 자주 바뀌는 정보는 모델에 외우게 하면 위험하다. 이 정보는 RAG로 문서에서 가져와야 한다.

LoRA는 나중에 답변 말투, 출력 형식, 근거 기반 요약 습관을 학습시키는 데 쓰는 것이 맞다. QLoRA는 더 큰 모델을 제한된 메모리에서 실험해야 할 때 검토한다. 즉, 순서는 `RAG 안정화 -> 평가 데이터 축적 -> LoRA/QLoRA 실험`이다.

## 용어 메모

- Hugging Face: 공개 모델과 데이터셋을 공유하고 내려받을 수 있는 플랫폼이다.
- Transformers: Hugging Face 모델을 Python에서 실행하기 위한 대표 라이브러리다.
- Gemma: Google이 공개한 오픈 모델 계열이다.
- MLX: Apple Silicon에서 모델을 효율적으로 실행하기 위한 런타임이다.
- 양자화: 모델 가중치의 정밀도를 낮춰 메모리 사용량을 줄이는 방식이다.
- 프롬프트: LLM에게 전달하는 지시문과 입력 내용이다.
- MockMvc: Spring MVC API를 테스트할 때 사용하는 테스트 도구다.
- instruction tuning: 모델이 지시를 더 잘 따르도록 예시 데이터로 학습시키는 방식이다.
- 생성 모드: 검색 결과를 바탕으로 LLM이 답변 문장을 만드는 실행 방식이다.
