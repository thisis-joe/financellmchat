# 08. 6차 답변 품질 사고 흐름

6차의 출발점은 "검색은 좋아졌는데, 챗봇답게 말하고 있는가?"라는 문제였다.  
이전 단계에서는 문서 검색과 LLM 답변 생성이 중심이었다. 하지만 실제 화면에서 `ㅎㅇ`, `예금 종류`, `50대인데 추천해주세요` 같은 질문을 넣어 보니 답변이 엉뚱하게 흘렀다.

가장 큰 원인은 모든 입력을 RAG 검색으로 보낸 것이었다. `ㅎㅇ`는 인사말인데도 문서 검색으로 들어가 청년 주택드림 청약통장 같은 상품 문서가 잡혔다. `예금 종류`는 예금상품 분류를 물은 것인데 예금거래기본약관의 휴면예금 문장이 답변으로 나왔다. `50대 추천`은 추천 질문인데 숫자 `50`이 `50만원` 같은 문서 표현과 섞여 청년/청약 상품이 올라왔다. 이후 `ㅂ2`, `ㅂㅇ`, `ㅅㄱ`, `수고`, `잘가` 같은 짧은 대화 표현도 같은 문제를 일으킬 수 있다는 점을 확인했다.

이 문제는 LLM만 바꿔서 해결할 문제가 아니었다. 모델은 검색된 근거를 바탕으로 답변하기 때문에, 검색 전에 질문의 성격을 나누어야 했다. 그래서 `rag-service/app.py`에 직접 응답 라우팅을 추가했다. 인사, 감사, 종료 표현은 문서 검색으로 보내지 않고 자연스럽게 응답한 뒤 검색 가능한 질문 예시로 유도한다. 반대로 `안녕 펫 적금 혜택 알려줘`처럼 인사말이 섞였더라도 상품명과 금융 의도가 있으면 RAG 검색으로 보낸다.

추천 흐름도 과하게 만들지 않았다. 이 프로젝트는 금융 자문 서비스가 아니라 상품공시 기반 RAG 학습 프로젝트다. 그래서 나이와 목적을 간단히 감지하고, 조건에 맞지 않는 상품을 제외한 뒤 후보 3개와 이유를 보여주는 정도로 제한했다. 예를 들어 50대 추천에서는 청년도약계좌, 청년 주택드림 청약통장, 장병내일준비적금처럼 청년/군복무 조건이 강한 상품을 우선 제외한다.

검색 범위도 다시 조였다. MySQL에는 실험 과정에서 예금, 대출, 기타상품 요약 데이터가 일부 섞여 있을 수 있다. 하지만 현재 프로젝트의 명확한 주제는 `예금상품 > 적립식예금`이다. 그래서 Python RAG의 기본 문서 조회 범위를 이 카테고리로 제한했다. 범위를 좁히는 것은 기능 축소가 아니라 답변 신뢰도를 올리는 선택이었다.

상품명이 명시된 질문도 보강했다. `펫 적금 혜택`이라고 물으면 출처가 펫 적금으로 모여야 한다. 이전에는 의미가 비슷한 다른 적금 문서가 함께 섞일 수 있었다. 그래서 질문에서 감지한 상품명과 다른 상품 문서에는 점수 패널티를 주었다. 이 변경 뒤에는 `펫 적금 혜택 받으려면 뭘 해야 해?` 질문에서 펫 적금 문서만 출처로 남았다.

답변 길이와 모양도 다듬었다. MLX 모델이 답변을 만들 때 토큰 수가 부족하면 문장이 중간에 끊길 수 있었다. 그래서 실행 문서의 `HF_MAX_NEW_TOKENS` 권장값을 260으로 올렸다. 또한 프롬프트에는 굵은 글씨, 근거 번호, 장식 문자를 피하고 확인 내용을 짧은 bullet로 쓰도록 지시했다.

검증은 거창한 평가 시스템이 아니라 대표 질문으로 했다. `ㅎㅇ`는 인사로, `예금 종류`는 분류 안내로, `50대인데 추천해주세요`는 청년상품 제외 추천으로, `펫 적금 혜택 받으려면 뭘 해야 해?`는 펫 적금 출처만 사용하는 답변으로 확인했다. 이후 이 기준을 `scripts/check_representative_questions.py`로 고정했다. 지금 단계에서는 이 정도의 작고 명확한 회귀 질문이 더 중요하다.

6차의 결론은 "좋은 RAG는 무조건 검색을 많이 하는 것이 아니라, 검색이 필요한 질문과 그렇지 않은 질문을 구분하는 것"이다. 프로젝트를 복잡하게 키우기보다 대표 사용 흐름을 안정화하는 것이 우선이다.

## 다음 과제

앞으로는 기능을 많이 늘리기보다 대표 질문 10개를 회귀 테스트처럼 관리한다. 현재 대표 질문은 인사말, 짧은 무의미 입력, 예금 종류, 적립식예금 상품 목록, 50대 추천, 만 50세 추천, 펫 적금 혜택, 장병내일준비적금 서류, 청년도약계좌 가입대상, 주택청약종합저축 가입자격이다.

새 PDF를 추가할 때도 관리자 화면을 만들기보다 `data/raw`에 넣고 importer를 실행하는 현재 방식을 유지한다. LoRA와 QLoRA는 지금 당장 적용하기보다 피드백 데이터가 쌓인 뒤 답변 형식과 말투를 학습시키는 실험으로 남겨 둔다.

## 용어 메모

- 질문 라우팅: 질문을 받은 뒤 어떤 처리 경로로 보낼지 먼저 결정하는 과정이다.
- 직접 응답: 문서 검색 없이 미리 정한 규칙으로 바로 답하는 방식이다. 인사말이나 범위 안내에 적합하다.
- 대화성 입력: `ㅂㅇ`, `수고`, `잘가`처럼 문서 검색보다 대화 응답이 먼저 필요한 짧은 표현이다.
- 검색 유도: 현재 적재된 문서 범위 안에서 답할 수 있도록 사용자의 질문을 예시와 함께 안내하는 방식이다.
- 추천 전용 흐름: 검색 점수만 보지 않고 나이, 목적, 제외 조건을 보고 후보를 고르는 처리다.
- 연령 필터: 질문에 나온 나이대와 맞지 않는 청년/장병/아동 전용 상품을 추천에서 제외하는 규칙이다.
- 상품 불일치 패널티: 질문에 특정 상품명이 있을 때 다른 상품 문서의 검색 점수를 낮추는 보정이다.
- 회귀 질문: 기능 개선 후에도 이전에 중요했던 질문이 계속 잘 답변되는지 확인하기 위한 대표 질문이다.
- 배포 검증: 로컬이 아니라 실제 공개 API 주소에 같은 질문을 보내 외부 접속 경로까지 확인하는 작업이다.
